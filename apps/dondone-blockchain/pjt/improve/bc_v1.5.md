# DonDone — 블록체인 파트 분리본

## 1. 블록체인 파트 역할 정의

DonDone에서 블록체인은 단순 전송 기능이 아니라, 아래 3가지 역할을 담당합니다.

1. **테스트넷 송금**

   * 등록된 수신자에게 스테이블코인(또는 모의 토큰) 전송
   * 전송 상태 추적
   * 전송 해시 기반 영수증 생성

2. **SafePay 기반 송금 보호**

   * 허용 목록
   * 24시간 쿨다운
   * 고액 추가 확인
   * 중복 전송 방지

3. **문서 무결성 보조**

   * Proof Pack / Claim Kit 해시 생성
   * QR 검증 연계
   * 문서가 위변조되지 않았음을 보여주는 데모 장치

---

## 2. 블록체인 범위(P0 / P1)

### P0 (6주 Demo MVP)

* 테스트넷 송금
* 송금 상태 추적
* 전송 해시(Tx Hash) 표시
* 송금 영수증 PDF 생성
* SafePay 최소 정책 적용

  * 허용 목록
  * 수신자 변경 후 24시간 송금 제한
  * 고액 추가 확인
  * 중복 방지 키
* 문서 해시 생성
* QR 검증 화면 연동
* 데모 시드 기반 송금 성공/실패 케이스 재현

### P1 (후속)

* 정기 송금/자동 송금
* 송금 한도 고도화
* 메인넷 연동
* 실제 Vault/Advance 사업 구조와의 연결
* 실사용 수준의 리스크 정책 및 온체인 운영 고도화

## 3. 핵심 사용자 플로우 중 블록체인 관련 부분

### 3.1 송금(테스트넷)

1. 수신자 선택(허용 목록 내)
2. 금액 입력
3. 사용자 확인
4. 테스트넷 전송 실행
5. 상태 추적

   * `SUBMITTED`
   * `CONFIRMED`
   * `FAILED`
6. 송금 영수증 PDF 생성 및 공유

### 3.2 SafePay 차단 흐름

* 허용 목록 외 수신자 송금 차단
* 수신자 변경 직후 24시간 송금 제한
* 고액 송금 시 추가 확인
* 중복 요청 시 1회만 처리

### 3.3 문서 무결성 검증

1. Proof Pack / Claim Kit 생성
2. 문서 payload 해시 생성
3. QR payload 포함
4. 검증 화면에서 `OK / FAIL` 표시

## 4. 기능 요구사항 — 블록체인 관련

# 4A. 송금(Remittance, 테스트넷)

### R1. 수신자 허용 목록(P0)

* 별칭/관계/사진(선택) 저장 가능
* 허용 목록 외 송금 불가
* 자주 보내는 사람 표시 지원 권장

### R2. 송금 실행(P0)

* 테스트넷에서 스테이블코인 또는 모의 토큰 전송
* 전송 전 사용자 확인 필수
* 화면에 반드시 표시:

  * 허용 목록 수신자
  * 데모 송금(실자금 이동 없음)
  * 영수증 저장 예정

### R3. 상태 추적(P0)

* 상태: `SUBMITTED -> CONFIRMED / FAILED`
* 실패 시 재시도 안내
* 송금 상세 화면에서 상태와 Tx Hash 재확인 가능해야 함

### R4. 영수증(P0)

* Tx Hash 포함
* PDF 영수증 자동 생성
* 송금 상세/내역 화면에서 즉시 열 수 있어야 함

---

# 4B. SafePay 정책

### 필수 정책(P0)

* 허용 목록 기반 송금
* 수신자 변경 후 24시간 송금 제한
* 고액 추가 확인

  * 예: 평소의 2배
  * 또는 특정 금액 이상
* 중복 전송 방지(Idempotency Key)

### UX 요구사항

* 차단 시 반드시 설명 필요:

  * 왜 막혔는지
  * 언제 다시 가능한지
  * 무엇을 해야 하는지

---

# 4C. 보관/이자(Vault, 데모 시뮬레이션)

### 블록체인 관점 역할

* 테스트넷 기반 잔액 흐름 체험
* 스테이블코인 보관/전환 시나리오 제공
* 실제 수익 서비스가 아닌 데모 시뮬레이션

### P0

* 현재 보관 금액 표시
* 보관 가능 금액 표시
* 송금 가능 전환 금액 표시
* 예상 이자 표시(일/월, 연환산 기준값)

### 유의사항

* 실제 수익 보장처럼 보이면 안 됨
* 메인넷 연동은 P1

## 5. 블록체인 아키텍처 요구사항

## 5.1 인터페이스 추상화

블록체인 로직은 adapter 뒤에 숨기고, 다른 팀원은 UseCase만 사용합니다.

```java
public interface TransferUseCase {
  TransferId requestTransfer(RequestTransferCommand cmd); // Idempotency-Key 포함
  TransferDto getTransfer(TransferId id);
}

public interface Erc20Gateway {
  TxHash submitTransfer(String tokenAddress, String toAddress, BigInteger amountAtomic);
  Optional<TxReceipt> getReceipt(TxHash txHash);
}
```

### 설계 원칙

* web3j 사용 여부는 adapter 내부에 한정
* nonce/gas/RPC 처리도 adapter 내부 책임
* 서비스 레이어는 송금 요청/조회만 알면 됨

---

## 5.2 비동기 잡 구조

송금은 요청 즉시 끝나는 기능이 아니라, **전송 + 확정 확인**이 분리되어야 합니다.

### 잡 종류

* `SUBMIT_TRANSFER`
* `POLL_TRANSFER_RECEIPT`
* `RENDER_TRANSFER_RECEIPT`

### 상태

* `QUEUED -> RUNNING -> DONE / FAILED`

### 이유

* 네트워크 재시도 필요
* 영수증 확정까지 시간차 존재
* 중복 요청 가능성 높음
* nonce 충돌 방지 필요
* MVP에서는 송금 잡 동시성 1로 직렬화하는 것이 안전

---

## 5.3 상태 머신

송금 상태는 최소 아래처럼 관리합니다.

```text
REQUESTED
  ->
SUBMITTED
  ->
CONFIRMED / FAILED
```

권장 확장:

```text
REQUESTED -> QUEUED -> SUBMITTED -> CONFIRMED
                              └-> FAILED
```

## 8. 문서 무결성 / 해시 / QR 검증

### 목적

* 생성된 문서가 위변조되지 않았음을 시연
* “근거 문서”의 신뢰 보강

### P0 요구사항

* Proof Pack 해시 생성(오프체인)
* QR payload 확정
* QR Verifier 웹에서 검증 결과 표시

  * `OK`
  * `FAIL`
* 생성일 / 포함 건수 / 해시 표시

### 포함 대상

* Proof Pack
* Claim Kit
* 필요 시 송금 영수증

## 9. 데모 모드에서의 블록체인 요구사항

### Time Travel 연동

* 기준일(`asOf`)에 따라 송금 상태도 자연스럽게 바뀌어야 함
* 성공 케이스, 실패 케이스 각각 재현 가능해야 함

### 데모 시드 데이터 포함 요소

* 정상 송금 1건
* 실패 송금 1건 이상
* 차단 케이스 1건

  * allowlist 위반
  * cooldown 위반
  * 고액 확인 필요
* 문서 해시 검증 성공 케이스
