# PRD v1.4 — WorkProof Pay

> **“벌고 → 지키고 → 보내는” 외국인 노동자의 급여·증빙·송금 파이프라인**
> 
> 
> 출퇴근 기록을 **증거로 남기고**, 급여는 **’정답 계산’이 아니라 이상 징후 탐지**로 점검하며, 문제가 생기면 **증빙(PDF)·신고 준비**까지 즉시 연결하고, 남은 돈은 **스테이블코인(테스트넷) 송금**으로 가족에게 빠르게 보내는 앱
>

## 0. 처음 보는 사람을 위한 설명

### 0.1 한 문장 요약

> **“출퇴근 기록을 증거로 남기고, 급여 이상 징후를 근거와 함께 보여주며, 버튼 한 번으로 증빙(PDF)과 신고 준비까지 만들고, 남은 돈은 스테이블코인으로 가족에게 빠르게 보내는 앱.”**
> 

### 0.2 해결하려는 문제

외국인 노동자는 다음 문제가 동시에 발생한다.
- **근무 증거가 약함**: 일한 기록이 흩어져 있어 나중에 증명하기 어렵다.
- **급여가 맞는지 확인이 어려움**: 연장·야간·공제 등으로 “받아야 할 돈”과 “받은 돈”이 다를 수 있는데, 무엇이 누락됐는지 알기 어렵다.
- **행동(항의/신고) 장벽**: 절차가 복잡하고 증빙 준비가 어렵다.
- **송금 불편**: 소액 송금은 비용 부담이 크고, 영수증/내역이 흩어진다.

### 0.3 우리가 택한 원칙(리스크 관리)

- “수정 불가”가 아니라 **수정 가능 + 감사 로그(설명 가능한 기록)**로 신뢰를 만든다.
- 급여 계산은 “정답 계산기”가 아니라 **이상 징후 탐지 + 근거 제시**로 범위를 고정한다.
- 인공지능은 “자유 대화”가 아니라 **화면 맥락 기반 코파일럿(설명/번역/문장 생성)**으로 제한한다.

## 1. 타깃 사용자

### 1.1 MVP 타깃

- 제조/건설/농축산/서비스 등 현장 중심 업종의 외국인 근로자
- 시급/일급/월급 혼재, 연장/야간 가능성 존재
- 한국어 문서(계약서/급여명세서) 이해가 어려움
- 매월 소액 송금이 잦음

### 1.2 핵심 사용자 목표(JTBD)

1. **내가 일한 기록을 확실히 남기고 싶다.**
2. **급여가 이상한지(누락/지연/공제 과다)를 쉽게 알고 싶다.**
3. 문제가 생기면 **증빙을 자동으로 만들어 바로 행동하고 싶다.**
4. 남는 돈을 **가족에게 빠르고 저렴하게 보내고 싶다.**

## 2. 목표 및 성공 기준

### 2.1 Demo MVP에서 반드시 보여줄 핵심(6-step)

1. **근무 기록 생성**: 출근/퇴근 기록 1건 생성
2. **월간 요약**: 이번 달 근무시간/연장/야간 요약 표시
3. **급여 점검**: 실제 입금액 입력 → 차액(이상 징후) 감지
4. **증빙 생성**: Proof Pack 생성(빠르게 생성 연출)
5. **송금**: 테스트넷 송금 → 상태 추적 → 영수증 표시
6. **신고 준비**: Claim Kit 생성 + Instant Claim v0(반자동 제출) + QR 무결성 검증 + 코파일럿 설명

### 2.2 데모 품질 목표(발표 임팩트)

- “한 달 사용자 여정”을 **15~30초로 압축**해 보여주기 위해, **타임트래블(데모 전용 기능)**을 사용한다.
- 차액 감지 화면에서 **다음 행동(증빙/메시지/신고 준비)**으로 끊김 없이 이어진다.

## 3. 범위(P0/P1) 확정표

> 원칙: P0는 **확실히 되는 데모**, P1은 **고도화**.
> 
> 
> P0에서 불확실성이 커지면 과감히 P1로 미룬다.
> 

| 영역 | P0(3주 데모 완주) | P1(후속) |
| --- | --- | --- |
| 근무 기록 | 출퇴근 원탭, **수정 사유 필수**, 첨부(사진/메모) 선택, 수정 이력 자동 저장/문서 반영 | 지오펜스 기반 알림 고도화, 다중 근무지 |
| 급여 점검 | **참고용 추정 + 이상 징후 탐지**(근거 포함), 공제 미입력 배지/임계값 완화 | 급여명세서/계약서 자동 추출(문서 파싱) 고도화 |
| 문서(PDF) | Proof Pack/Claim Kit/송금 영수증 PDF(템플릿 v1) | 템플릿 다국어/확장 |
| 신고 준비 | Instant Claim v0(반자동): 안내/체크리스트/자동 문장/공유/바로가기 | 기관 제출 자동화 또는 제휴 연동 |
| 송금 | 테스트넷 전송, 상태 추적, SafePay(허용 목록/쿨다운/중복 방지/고액 확인) | 정기/자동 송금(사용자 확인 포함), 한도 고도화 |
| 코파일럿(AI) | 화면 설명/번역/문장 생성 3종 + facts-only 가드레일 | FAQ/용어집 확장, 언어 확대 |
| RAG | 필수 아님, 필요 시 **RAG-lite(FAQ/용어집)**만 | 전면 챗봇은 P1 이후 |
| 데모 기능 | 타임트래블(시드 데이터 + asOf 렌더링) | 데모 시나리오/데이터 확장 |
| 무결성 | Proof Pack 해시 + QR 포함 + 검증 화면(✅/❌) | 서명/장기검증 고도화 |

## 4. 핵심 용어

| 용어 | 설명 |
| --- | --- |
| 근무 증거(WorkProof) | 출퇴근 시점의 시간/위치 스냅샷 + 사용자 확인으로 만든 근무 기록 1건 |
| 급여 점검 | 참고용 급여 추정과 실제 입금을 비교해 “이상 징후”를 잡고 근거를 제시하는 기능 |
| 증빙 리포트(Proof Pack) | 근무 로그 + 급여 근거 + 차액 리포트를 묶은 PDF |
| 신고 꾸러미(Claim Kit) | 제출/상담을 위해 필요한 요약·근거·첨부를 묶은 문서(PDF/ZIP) |
| 신고 준비(Instant Claim v0) | 자동 제출이 아니라 **제출을 쉽게 만드는 반자동 지원** |
| 허용 목록 | 송금은 등록된 수신자에게만 가능하도록 제한하는 목록 |
| 중복 방지 키 | 같은 요청이 여러 번 와도 1번만 처리하도록 하는 키(송금/문서 생성에 필수) |
| 전송 해시 | 블록체인 전송의 고유 ID(상태 추적/영수증에 포함) |
| 타임트래블(asOf) | 기준일을 바꾸면 그 날짜까지의 데이터만 반영해 화면이 바뀌는 **데모 전용 장치** |

## 5. 핵심 사용자 플로우

### 5.1 최초 1회: 기본 설정

1. 근무지 등록
2. 급여 정보 등록: **기본급 1줄 + 단위 선택(시급/일급/월급)**
3. 송금 수신자 등록(허용 목록)

### 5.2 매일: 출근/퇴근(근무 증거 생성)

- 출근 탭 → 기록 생성
- 퇴근 탭 → 기록 확정
- 필요 시 근무 시간 수정(W4) → **수정 사유 필수 + 증거 첨부 선택 + 수정 이력 자동 기록**

### 5.3 매월: 급여 점검

1. 월간 요약 확인
2. 참고용 급여 추정 확인(“참고/추정” 표시)
3. 실제 입금액 입력
4. 결과: 정상(허용 범위) / 이상(차액 감지 + 근거)

### 5.4 차액 감지 후 액션(가장 중요한 플로우)

차액 감지 화면에서 한 번에 제공:
1) **증빙 리포트 만들기(Proof Pack)**

2) **사업주 메시지 초안(다국어)**

3) **신고 꾸러미 만들기(Claim Kit)**

4) **신고 준비(Instant Claim v0)**: 제출 경로 안내 + 자동 요약 문장 + 체크리스트 + 파일 공유 + 바로가기

> UX 목표: “차액 감지 → 다음 행동”이 1 화면에서 끊기지 않게 연결된다.
> 

### 5.5 송금(테스트넷)

1. 수신자 선택(허용 목록)
2. 금액 입력 → 확인
3. 전송 후 상태 추적(SUBMITTED → CONFIRMED/FAILED)
4. 영수증 PDF 생성 및 공유

## 6. 기능 요구사항(상세)

## 6A. 근무 증거(WorkProof)

### W1. 근무지 등록(P0)

- 이름, 주소, 지도 선택

### W2. 급여 입력 UI 단순화(P0)

- **기본급 1줄 + 단위 선택(원/시간, 원/일, 원/월)**로 통일
- 단위별 조건 입력(1개만 추가 노출)
    - 일급: 하루 기준 근무시간(기본 8시간, 수정 가능)
    - 월급: 월 기준 근무시간(기본값 제공, 수정 가능)
- 내부 로직은 **시간당 기준으로 정규화(환산 시급)**하여 집계/계산 엔진 재사용

### W3. 출퇴근 원탭 기록(P0)

- 출근/퇴근 시점 위치 스냅샷 1회 저장(백그라운드 추적 없음)
- 기기 시간 + 서버 수신 시간 함께 기록(감사 목적)

### W4. 근무 시간 수정 — ‘증빙 강화 UX’(P0)

**목표:** “수정 가능 = 조작”이 아니라 **“수정 가능 = 설명 가능한 기록”**으로 신뢰 강화

### W4-1 수정 사유 선택(필수)

- 시간 수정 시 수정 사유 선택을 반드시 받는다.
- 초기 사유(예시):
    - 출근/퇴근 버튼을 늦게 눌렀어요
    - 연장근무가 있었어요
    - 휴게시간이 달랐어요
    - 기록이 누락됐어요
    - 기타(직접 입력)

### W4-2 추가 증거 첨부(선택)

- 사진/메모 첨부 제공(없어도 수정 가능)
- 첨부가 없을 경우에도 “첨부 없음”을 명확히 기록한다.

### W4-3 수정 이력(감사 로그) 자동 기록(필수)

- 누가: 사용자 ID
- 언제: 서버 기준 수정 시각
- 무엇을: 변경 전/후 값(시간/관련 필드)
- 왜: 수정 사유(선택값 + 기타 메모)
- 증거 유무: 첨부 개수/종류 + 파일 참조(식별자)

### W4-4 문서 자동 반영(필수)

- Proof Pack/Claim Kit에는 수정된 기록 표시(배지)와 수정 이력 표가 자동 포함된다.
- 첨부가 있으면 첨부 목록(종류/업로드 시각/식별값)이 자동 포함된다.

### W5. 월간 요약(P0)

- 월간 근무일/총 근무시간/연장/야간 요약
- “수정된 기록 수” 요약 포함

## 6B. 급여 점검 — ‘정답 계산’이 아니라 ‘이상 징후 탐지’

### 필수 안내 문구(화면 하단 고정)

- “이 급여 계산은 **참고용 추정**입니다. 실제 지급/공제는 근로계약, 급여명세서, 회사 규정에 따라 달라질 수 있습니다.”
- 공제 미입력 시: “공제가 입력되지 않아 **공제가 반영되지 않았습니다.**(공제 미반영)”

### WS1. 참고용 급여 추정(P0)

- WorkProof 집계 결과로 추정(근거 표시 필수)
- 중복 계산 방지(프리미엄은 추가분만)
    - 기본: 시급 × 총 근로시간 × 1.0
    - 연장 프리미엄: 시급 × 연장시간 × 0.5
    - 야간 프리미엄: 시급 × 야간시간 × 0.5
- 근거 표시:
    - 시간/구간/WorkProof ID 목록
    - 수정 사유/첨부 유무 배지

### WS2. 실제 입금액 입력(P0)

- 기본은 “통장 입금액(실수령)” 입력
- 공제 항목 미입력 시:
    - “공제 미반영” 배지 표시
    - 알림 임계값 완화(불필요 경보 최소화)

### WS3. 차액 감지(이상 징후 탐지) (P0)

- 임계값(기본):
    - 차액 ≥ 30,000원 또는 예상액의 2%
    - 공제 미반영이면 50,000원 또는 3%
- 차액 화면은 “가능한 원인”을 근거와 함께 제시한다.
    - 예: 연장시간 합계, 야간시간 합계, 수정된 기록 포함 여부, 근거 WorkProof 목록

## 6C. 송금(테스트넷) + SafePay

### R1. 수신자 허용 목록(P0)

- 허용 목록 외 송금 불가
- 수신자 정보: 별칭/관계/사진(선택)

### R2. 송금 실행(P0)

- 전송 전 확인 화면 필수
- 중복 방지 키로 중복 전송 차단

### R3. 상태 추적(P0)

- SUBMITTED → CONFIRMED / FAILED
- 실패는 원인을 분류해 화면에 표준화된 코드로 보여준다(“네트워크 오류”, “정책 차단” 등)

### R4. 영수증(P0)

- 전송 해시 포함
- 영수증 PDF 생성 및 공유

### SafePay 정책(P0)

- 허용 목록만 송금 가능
- 수신자 변경 후 24시간 쿨다운
- 고액 추가 확인(상대/절대 기준은 오픈 이슈에서 확정)
- 중복 방지 키(서버/DB 레벨)

---

## 6D. 문서(PDF) + 신고 준비

### D1. Proof Pack(P0)

필수 포함:
- 월간 요약 + WorkProof 상세 표
- 급여 추정 근거 + 차액 리포트
- **수정 이력(감사 로그) 표**
- **첨부 목록(있으면)**

- 무결성 섹션(해시/생성 시각/포함 건수) + QR

### D2. Claim Kit(P0)

- Proof Pack + 제출용 요약 + 제출 체크리스트
- 출력 형태:
    - PDF(기본)
    - 첨부가 많으면 ZIP(선택)

### D3. 신고 준비(Instant Claim v0) (P0)

- 자동 제출이 아니라 반자동 지원(문구 고정):
    - “이 기능은 제출을 대신하지 않습니다. 제출은 사용자가 직접 진행합니다.”
- 제공 기능:
    - 접수 경로 안내(온라인/전화/방문)
    - 제출용 요약 문장(복사 버튼)
    - Claim Kit 공유(메일/메신저/저장)

## 7. 코파일럿(AI) 설계 — 화면 맥락 보조 기능

### 7.1 원칙

- 코파일럿은 **설명/번역/문장 생성만** 한다.
- 숫자 계산/판정은 **서버의 룰 기반 결과(facts)**로 고정한다.
- 코파일럿은 facts 밖의 내용을 추정하거나 새로 계산하지 않는다.

### 7.2 코파일럿 UX(버튼/추천 질문 칩)

### (1) 차액 감지 화면

- 버튼: `이 화면 설명해줘`
- 칩 예시:
    - `왜 차액이 생겼어?`
    - `어떤 근거가 있어?`
    - `지금 무엇을 하면 돼?`

### (2) SafePay 차단 화면

- 버튼: `왜 송금이 막혔어?`
- 칩 예시:
    - `쿨다운이 뭐야?`
    - `고액 확인 기준이 뭐야?`
    - `언제 다시 시도할 수 있어?`

### (3) 신고 준비 화면

- 버튼: `제출용 문장 만들어줘`
- 칩 예시:
    - `정중하지만 단호하게`
    - `짧게 요약`
    - `영어로 번역`(P0), `베트남어 번역`(P1)

### 7.3 코파일럿 API(권장 3종, P0)

1. 차액/정책 차단 이유를 쉽게 설명(근거 포함)
2. 사업주 메시지 초안 다국어 생성(정중+단호, 근거 포함)
3. 제출용 요약 문장 생성(반자동 제출 지원)

### 7.4 가드레일(환각/오답 방지) — PRD 고정

- 코파일럿은 **숫자를 새로 계산하거나 추정하지 않는다.**
- 코파일럿 입력은 서버가 만든 **facts JSON**만 사용한다.
- facts 예시:
    - 차액 금액, 근무시간 합계, 연장/야간 시간, WorkProof ID 목록, 수정 사유 목록, SafePay 차단 코드 등
- facts 밖 질문에는 “확인 불가/추정 불가”로 답하고 필요한 입력을 안내한다.
- 공통 문구:
    - “이 안내는 화면에 표시된 사실을 바탕으로 작성된 참고 문장입니다.”

### 7.5 RAG 정책

- 대부분 질문은 화면 facts로 해결한다.
- 용어/기능 설명만 위해 **RAG-lite(FAQ/용어집)**를 선택 적용한다.
- 전면 챗봇은 P1 이후로 미룬다.

## 8. 데모 전용 기능: 타임트래블(asOf)

> 실사용 기능이 아니라 발표 데모를 위한 장치이며, 릴리스 빌드에서는 비활성화한다.
> 

### 8.1 목적

- 한 달 흐름을 15~30초로 압축해 보여주기

### 8.2 동작 방식

- 데모 시작 시 **시드(고정) 데이터**를 주입한다.
    - 정상 케이스 1개, 차액 케이스 1개(의도된 누락 포함)
- 슬라이더/재생 버튼으로 기준일(asOf: YYYY-MM-DD)을 바꾸면,
    - 그 날짜 이전 데이터만 반영한 월간 요약/급여 점검/차액/액션이 즉시 렌더링된다.

### 8.3 구현 권장안

- 서버 데모 API
    - `POST /demo/seed` : 시드 데이터 주입
    - `POST /demo/reset` : 데모 상태 초기화
    - `GET /demo/state?asOf=YYYY-MM-DD` : 기준일 이전 데이터 기준으로 집계된 상태 반환
- 앱은 `X-Demo-AsOf` 헤더 또는 쿼리로 기준일을 전달한다.

## 9. 무결성(해시/QR) 설계 — P0

### 9.1 목적

- “문서가 나중에 바뀌지 않았음”을 데모에서 직관적으로 보여준다.

### 9.2 해시 대상(초안)

- Proof Pack 생성 시, 아래를 포함한 “정규화된 JSON”을 만든다.
    - 문서 메타: 문서 ID, 생성 시각, 생성 주체
    - 포함된 WorkProof ID 목록 + 각 기록의 핵심 필드(출근/퇴근 시간, 수정 이력 요약)
    - 차액 리포트의 핵심 필드(차액 금액, 근거 집계값)
    - 첨부 목록(파일 식별자)
- 해시 알고리즘: SHA-256(권장)

> P0에서는 “오프체인 해시”로 충분. 서명/장기 검증은 P1 이후.
> 

### 9.3 QR 페이로드(초안)

- QR에는 최소 정보를 담는다.
    - 문서 ID
    - 생성일
    - 포함 건수(WorkProof 개수)
    - SHA-256 해시
    - 검증 URL(선택)
- 검증 결과 화면(✅/❌)에서 표시:
    - 생성일, 포함 건수, 해시, 검증 결과

### 9.4 검증 방식(P0)

- 검증 웹(간단) 또는 앱 내 검증 화면에서:
    - 문서 ID 조회 → S3에서 문서/메타 조회 → 동일 규칙으로 해시 재계산 → 비교

### 10.4 잡(비동기) 설계(송금/문서 생성 안정화)

- PDF 생성과 송금 “확정 확인”은 잡으로 처리한다.
- 잡 상태: QUEUED → RUNNING → DONE / FAILED

잡 종류(예시):
- RENDER_PROOF_PACK, RENDER_CLAIM_KIT, RENDER_TRANSFER_RECEIPT
- SUBMIT_TRANSFER, POLL_TRANSFER_RECEIPT

### 10.5 송금이 비동기 처리인 이유(요약)

- 전송은 빨라도 “완료 확정”은 영수증 확인이 필요하다.
- 모바일 재시도/중복 요청이 흔해 중복 방지 키와 재시도 정책이 필요하다.
- nonce 충돌을 피하기 위해 전송 잡은 직렬화(동시성=1)가 MVP에 안전하다.

## 12. “미리 설계해야 하는 것” 체크리스트(PRD 포함)

### 12.1 표준 상태/코드(필수)

- 송금 상태: REQUESTED / SUBMITTED / CONFIRMED / FAILED
- SafePay 차단 사유 코드(프론트 표시와 1:1 매핑)
- 잡 상태: QUEUED / RUNNING / DONE / FAILED
- 공통 에러 코드/메시지(사용자용/개발자용 분리)

### 12.2 중복 방지 키 규칙(필수)

- 송금/문서 생성 API는 중복 방지 키를 받아 DB에서 유일성으로 차단한다.
- 동일 키 재요청은 “기존 결과” 반환

### 12.3 문서 페이로드(JSON) 규격(필수)

- Proof Pack/Claim Kit 템플릿이 안정적으로 렌더링되도록, 서버가 보내는 페이로드 스키마를 고정한다.

### 12.4 데모 시드 데이터 포맷(필수)

- 정상 케이스 1, 차액 케이스 1(의도된 누락 포함)
- WorkProof 수정 이력/첨부 포함

### 12.5 관측 가능성(필수)

- 요청 ID(requestId) 로그: 앱→서버→잡→pdf-service까지 추적 가능
- 핵심 이벤트(송금 요청/문서 생성/차액 감지) 로그 표준화
