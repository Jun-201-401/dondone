# PRD v1.4 — DawnDone
> **“당신의 돈(Don), 시작(Dawn)부터 끝(Done)까지 던던하게.”**  
> 근무 기록을 **‘증거로 쓸 수 있게’** 남기고, 급여를 **‘이상 징후 중심’**으로 점검하며, 필요하면 **증빙(PDF)·신고 준비**까지 바로 연결하고, 남은 돈은 **스테이블코인(테스트넷) 송금**으로 가족에게 빠르게 보내는 앱

## 0. 처음 보는 사람을 위한 설명

### 0.0 브랜드 표기 원칙(v1.4 추가)
- 서비스명 표기는 **DawnDone**을 기본으로 사용한다.
- 발표/데모 화면의 핵심 슬로건은 아래 문구를 고정 사용한다.
  - **“당신의 돈(Don), 시작(Dawn)부터 끝(Done)까지 던던하게.”**
- 기술 맥락에서 기존 프로젝트명(`WorkProofPay`)이 남아 있어도, 대외 노출 텍스트는 DawnDone 기준으로 통일한다.

### 0.1 한 문장 요약
> **“출퇴근 기록을 증거로 남기고, 급여 이상 징후를 찾아 근거와 함께 보여주며, 버튼 한 번으로 증빙(PDF)과 신고 준비까지 만들고, 남은 돈은 스테이블코인으로 가족에게 빠르게 보내는 앱.”**

### 0.2 이 제품이 해결하려는 핵심 문제
외국인 노동자는 다음 문제가 동시에 발생한다.
- **근무 증거 부족**: ‘일했다’는 기록이 약하거나 흩어져 있어 나중에 증명하기 어렵다.
- **급여 확인 어려움**: 연장·야간·공제 등으로 ‘내가 받아야 할 돈’과 ‘실제로 받은 돈’이 다를 수 있는데, 무엇이 누락됐는지 알기 어렵다.
- **행동(항의/신고) 장벽**: 절차가 복잡하고, 증빙 준비가 부담이라 중간에 포기하기 쉽다.
- **송금 불편**: 소액을 자주 보내면 수수료 부담이 커지고, 내역/영수증이 흩어진다.

### 0.3 우리가 만드는 해법의 핵심
- 기록을 “수정 불가”로 만드는 대신, **수정 가능하되 ‘설명 가능한 기록’**으로 만들어 신뢰를 올린다.
- 급여는 “정답 계산기”가 아니라 **이상 징후 탐지 + 근거 제시**로 범위를 조절해 정확도 리스크를 낮춘다.
- 인공지능은 “자유 대화 챗봇”이 아니라, **현재 화면 맥락을 설명하고 문장을 생성하는 보조 기능(코파일럿)**으로 최소 범위 도입한다.

## 1. 문제 정의

### 1.1 돈의 흐름에서 발생하는 문제(전체 구간)
```
일한다 ──→ 급여 받는다 ──→ 생활비 ──→ 남은 돈 송금
  │            │                          │
  ▼            ▼                          ▼
증거 약함     누락/체불 확인 어려움         수수료/속도/영수증 관리 번거로움
```

### 1.2 기존 대안의 한계
| 기존 방식 | 되는 것 | 안 되는 것 |
|---|---|---|
| 은행 해외송금 | 송금 | 비용/시간 부담, 근태·급여 검증과 연결 없음 |
| 노동부/노무사 | 상담/진정 | 증거를 사용자가 직접 준비해야 함 |
| 근태관리 앱 | 출퇴근 기록 | 급여 점검/증빙/신고 연결이 약함 |
| 개인 지갑 앱 | 코인 전송 | 안전정책·자금출처·증빙 자동화·외국인 UI 부족 |

## 2. 타깃 사용자

### 2.1 MVP 타깃
- 제조/건설/농축산/서비스 등 현장 중심 업종의 외국인 근로자
- 시급/일급/월급 혼재, 연장/야간 가능성 존재
- 한국어 문서(계약서/급여명세서) 이해가 어려움
- 매월 소액 송금이 잦음

### 2.2 핵심 사용자 목표(JTBD)
1) **내가 일한 기록을 확실히 남기고 싶다.**
2) **급여가 이상한지(누락/지연/공제 과다)를 쉽게 알고 싶다.**
3) **문제가 생기면 증빙을 자동으로 만들어 바로 행동하고 싶다.**
4) **남는 돈을 가족에게 빠르고 저렴하게 보내고 싶다.**

## 3. 제품 목표

### 3.1 Demo MVP에서 반드시 보여줄 것(P0)
- **근무 기록(증거화)**: 출근/퇴근 기록 + 수정 사유 필수 + 수정 이력 자동 저장
- **급여 점검(이상 탐지)**: 참고용 급여 추정 + 실제 입금 비교 + 차액 근거 제시
- **차액 이후 액션**: Proof Pack / 사업주 메시지 초안 / Claim Kit / 신고 준비(Instant Claim v0)
- **테스트넷 송금**: 수신자 목록(허용 목록) + 전송 + 상태 추적 + 영수증
- **문서(PDF)**: Proof Pack, Claim Kit, 송금 영수증을 S3에 저장하고 앱에서 공유/다운로드

### 3.2 성공 기준(데모 관점)
- 3분 데모에서 “한 달 여정”을 **15~30초로 압축해서 보여주는 데모 장치(Time Travel)**로 스토리를 완주
- “차액 감지 → 문서 생성 → 신고 준비”까지 **한 화면에서 자연스럽게 이어지는 흐름** 구현

## 4. 범위(P0/P1) 확정표

> 원칙: P0는 **확실히 되는 데모**, P1은 **발표 임팩트 강화/고도화**.  
> P0에서 불확실성이 커지면 과감히 P1로 미룬다.

| 영역 | P0(6주 Demo MVP) | P1(후속) |
|---|---|---|
| 근무 기록 | 출근/퇴근 원탭, 수정 사유 필수, 첨부(사진/메모) 선택, 수정 이력 자동 포함 | 지오펜스 기반 자동 알림 고도화, 다중 근무지 |
| 급여 점검 | **참고용 추정 + 이상 탐지**(근거 포함), 공제 미입력 배지/임계값 완화 | 급여명세서/계약서 자동 추출(문서 파싱) 고도화 |
| 문서(PDF) | Proof Pack/Claim Kit/송금 영수증 PDF(템플릿 v1) | 문서 템플릿 확장, 다국어 템플릿 |
| 신고 준비 | Instant Claim v0(반자동): 안내/체크리스트/자동 문장/공유/바로가기 | 기관 제출 자동화 또는 제휴 연동 |
| 송금 | 테스트넷 전송, 상태 추적, 안전정책(허용 목록/쿨다운/중복 방지) | 정기 송금/자동 송금(사용자 확인 포함), 한도 고도화 |
| 인공지능(코파일럿) | 화면 설명/번역/문장 생성 3종 + 가드레일(서버 facts만 사용) | RAG-lite 고도화, 언어 확장 |
| RAG | **필수 아님**. 기본은 화면 facts | 필요 시 FAQ/용어집 기반 RAG-lite |
| Vault(예금) | **P1**: 스테이블코인 예치 → DeFi yield 이자 시뮬레이션 + 목표 저축(테스트넷) | 메인넷 연동, 실 yield 파트너십 |
| Advance(가불) | **P1**: 근무 증거 기반 소액 가불(최대 ₩50만, Vault 풀 자금) 시뮬레이션 | 정책/규제 검토 후 상용화 |

## 5. 핵심 용어(이 문서에서 사용하는 말)

| 용어 | 설명 |
|---|---|
| 근무 증거(WorkProof) | 출퇴근 시점의 시간/위치 스냅샷 + 사용자 확인으로 만든 “근무 기록 1건” |
| 급여 점검(Wage Shield) | 참고용 급여 추정과 실제 입금을 비교해 “이상 징후”를 잡고 근거를 제시하는 기능 |
| 증빙 리포트(Proof Pack) | 근무 로그 + 계산 근거 + 차액 리포트를 한 파일(PDF)로 묶은 문서 |
| 신고 꾸러미(Claim Kit) | 제출/상담에 필요한 요약·근거·첨부를 묶은 문서(PDF 또는 ZIP) |
| 신고 준비(Instant Claim v0) | 신고를 “자동 제출”하는 기능이 아니라, **제출을 쉽게 만드는 반자동 지원** |
| 허용 목록 | 송금은 등록된 수신자에게만 가능하도록 제한하는 목록 |
| 중복 방지 키(Idempotency Key) | 같은 요청이 여러 번 와도 1번만 처리하도록 하는 요청 키(송금/문서 생성에 필수) |
| 전송 해시(Tx Hash) | 블록체인 전송의 고유 ID(상태 추적/영수증에 포함) |
| Vault(예금) | 유휴 스테이블코인을 예치하면 DeFi yield로 이자를 받는 구조(은행 적금과 유사). 풀의 일부는 Advance 자금으로 활용 |
| Advance(가불) | 근무 증거(WorkProof) 기반으로 급여일 전 소액(최대 ₩50만)을 먼저 받는 구조. 자금은 Vault 풀에서 제공, 급여일에 자동 상환 |
| 목표 저축(Goal Savings) | Vault 내에서 목표(학비/집/귀국)를 설정하면 급여에서 자동 적립, 달성 시 송금 연결 |

## 6. 핵심 사용자 플로우(특히 ‘차액 감지 후 액션’ 포함)

### 6.1 최초 1회: 기본 설정
1) 근무지 등록(이름/주소)
2) 급여 정보 등록: **기본급 1줄 + 단위 선택(시급/일급/월급)**  
3) 송금 수신자 등록(허용 목록)

### 6.2 매일: 출근/퇴근(근무 증거 생성)
- 출근 탭 → 기록 생성  
- 퇴근 탭 → 기록 확정  
- (필요 시) 근무 시간 수정(W4) → **수정 사유 필수 + 추가 증거 선택 첨부**

### 6.3 매월: 급여 점검
1) 월간 요약 확인(근무일/근무시간/연장/야간)
2) 참고용 급여 추정 확인(“참고/추정” 표기)
3) 급여일 이후 **실제 입금액 입력**
4) 결과:
   - 정상: “차이 없음(또는 허용 범위)”  
   - 이상: “차액 감지” + “근거 제시”

### 6.4 차액 감지 후 액션(가장 중요한 플로우)
차액 감지 화면에서 한 번에 제공:
1) **증빙 리포트 만들기(Proof Pack)**  
2) **사업주 메시지 초안 생성(다국어)**  
3) **신고 꾸러미 만들기(Claim Kit)**  
4) **신고 준비(Instant Claim v0)**: 제출 경로 안내 + 자동 요약 문장 + 체크리스트 + 파일 공유 + 바로가기

> 핵심 UX 목표: “차액 감지 → 다음 행동”이 1 화면에서 끊기지 않게 연결된다.

### 6.5 송금(테스트넷)
1) 수신자 선택(허용 목록)
2) 금액 입력 → 확인
3) 전송 후 상태 추적(SUBMITTED → CONFIRMED/FAILED)
4) 영수증 PDF 생성 및 공유

## 7. 기능 요구사항(상세)

## 7A. 근무 증거(WorkProof)

### W1. 근무지 등록(P0)
- 이름, 주소, 지도 선택

### W2. 급여 정보 입력(P0, 단순 UI)
- 입력 UI는 **“기본급 1줄 + 단위 선택(원/시간, 원/일, 원/월)”**로 통일
- 단위별 조건 입력(1개만 추가 노출)
  - 일급: 하루 기준 근무시간(기본 8시간, 수정 가능)
  - 월급: 월 기준 근무시간(기본값 제공, 수정 가능)
- 내부 로직은 **시간당 기준으로 정규화(환산 시급)**하여 동일 엔진 재사용

### W3. 출퇴근 원탭 기록(P0)
- 출근/퇴근 시점에 위치 스냅샷 1회 저장(백그라운드 추적 없음)
- 기기 시간 + 서버 수신 시간 함께 기록(감사 로그 기반)

### W4. 근무 시간 수정(‘증빙 강화 UX’로 설계) (P0)
**목표:** “수정 가능 = 조작”이 아니라 **“수정 가능 = 설명 가능한 기록”**으로 신뢰 강화

#### W4-1 수정 사유 선택(필수)
- 사용자가 시간을 수정할 때 **수정 사유 선택이 필수**
- 예시(초기 제공):
  - 출근/퇴근 탭을 늦게 눌렀어요
  - 연장근무가 있었어요
  - 휴게시간이 달랐어요
  - 기록이 누락됐어요
  - 기타(직접 입력)

#### W4-2 추가 증거 첨부(선택)
- 사진(예: 작업지시서, 근무표, 현장 사진 등) / 메모 첨부 가능
- 첨부는 선택이며, “첨부 없음”도 명확히 기록

#### W4-3 수정 이력(감사 로그) 자동 저장
- 수정 시 아래 정보를 **자동으로 기록**
  - 누가: 사용자 ID(또는 계정)
  - 언제: 수정 시각(서버 시간)
  - 무엇을: 변경 전/후 시간(또는 변경된 필드)
  - 왜: 수정 사유(선택값 + 기타 메모)
  - 증거 유무: 첨부 개수/종류(사진/메모) 및 파일 참조
- 수정 이력은 **삭제 불가(감사 목적)**, 단 사용자는 “추가 메모”는 보완 가능(기록으로 남김)

#### W4-4 문서 자동 반영 명세
- Proof Pack/Claim Kit에는 다음이 자동 포함되어야 함:
  - 수정된 기록 표시(배지)
  - 수정 이력 표(변경 전/후, 사유, 첨부 유무, 시각)
  - 첨부가 있는 경우: 첨부 목록(파일명/업로드 시각/해시 또는 식별값)

### W5. 월간 요약(P0)
- 월간 근무일, 총 근무시간, 연장/야간 시간 요약
- “수정된 기록 수” 요약 포함(증빙 신뢰 강화)

---

## 7B. 급여 점검(Wage Shield) — ‘정답 계산기’가 아니라 ‘이상 탐지’

### 핵심 원칙(문구/디스클레이머)
- 급여 계산 결과는 **“참고용 추정”**이며, 실제 지급/공제는 근로계약·법규·급여명세서에 따라 달라질 수 있다.
- 앱은 **이상 징후를 빠르게 찾고 근거를 보여주는 도구**이며, 법률 자문이 아니다.

### WS1. 참고용 급여 추정(P0)
- 입력된 급여 정보(시급/일급/월급 환산) + WorkProof 기반 시간 집계로 추정
- 중복 계산 방지(프리미엄은 추가분만)
  - 기본: 시급 × 총 근로시간 × 1.0
  - 연장 프리미엄: 시급 × 연장시간 × 0.5
  - 야간 프리미엄: 시급 × 야간시간 × 0.5
- 화면에는 항상 근거를 함께 표시:
  - 시간/구간/WorkProof ID
  - 수정 사유 유무(수정 배지)

### WS2. 실제 입금액 입력(P0)
- 사용자는 “통장 입금액(실수령)”을 입력하는 것을 기본으로 한다.
- 공제 항목을 입력하지 않은 경우:
  - **“공제 미반영” 배지** 표시
  - 알림 임계값을 완화(불필요 경보 최소화)

### WS3. 차액 감지(이상 징후 탐지) (P0)
- 임계값(기본값)
  - 차액 ≥ 30,000원 또는 예상액의 2%
  - 공제 미반영이면 50,000원 또는 3%로 완화
- 차액 화면에서 “가능한 원인”을 **근거와 함께** 제시(예: 연장시간 합계, 야간시간 합계, 수정 기록 포함 여부)

### WS4. 근거 중심 출력(P0)
- “총 차액”만 보여주지 않고, **항목별 근거**를 제공:
  - 연장/야간 합계 시간
  - 해당 WorkProof 목록(링크)
  - 수정 사유/첨부 유무 표시

### WS5. 문서 파싱 고도화(P1)
- 급여명세서/계약서 자동 추출(후속)
- MVP 단계에서는 수동 입력이 기본이며, 파싱은 “있으면 편한 기능”으로 제한

---

## 7C. 송금(Remittance, 테스트넷)

### R1. 수신자 허용 목록(P0)
- 별칭/관계/사진(선택) 포함
- 허용 목록 외 송금 불가

### R2. 송금 실행(P0)
- 테스트넷에서 스테이블코인(예: USDC 또는 모의 토큰) 전송
- 전송 전 반드시 사용자 확인(확인 화면)

### R3. 상태 추적(P0)
- 상태: SUBMITTED → CONFIRMED / FAILED
- 실패 시 재시도 안내(정책 기반)

### R4. 영수증(P0)
- 전송 해시(Tx Hash) 포함
- 영수증 PDF 자동 생성(문서 모듈 재사용)

## 7F. Vault — 스테이블코인 예금 (P1)

**"안 쓰는 돈을 넣으면 이자가 나온다 — 은행 적금처럼, 하지만 더 높은 이율로"**

> 유휴 스테이블코인(USDC)을 Vault에 예치하면 DeFi 프로토콜(Aave, sDAI 등)에서
> 연 3~5% 이자 수익이 발생한다. Vault 풀의 일부는 Advance(가불) 자금으로 활용되어 추가 수익을 만든다.
> MVP에서는 테스트넷 시뮬레이션으로 시연한다.

### 자금 순환 구조
```
┌─────────────────────────────────────────────────────┐
│                  Vault 풀 (USDC)                     │
│                                                     │
│  사용자들의 예치금                                     │
│      │                                               │
│      ├── 70% → DeFi 프로토콜 (Aave/sDAI)             │
│      │         연 3~5% yield                         │
│      │                                               │
│      └── 30% → Advance 풀 (가불 자금)                │
│               가불 수수료 수익 (₩1,500/건)            │
│                                                     │
│  수익 → 예치자에게 이자로 분배                         │
└─────────────────────────────────────────────────────┘
```

### V1. Vault 예치 (P1)
- 스테이블코인(USDC) 예치 → DeFi yield 자동 적용
- MVP에서는 고정 yield 시뮬레이션(예: 연 4.3%)

### V2. 이자 수익 대시보드 (P1)
- 일별/월별 이자 수익 표시
- 풀 현황: DeFi 운용 비율 / 가불 활용 비율

### V3. 목표 저축 (P1)
- 목표(학비/집 수리/귀국 자금) 설정 + 급여에서 자동 적립
- 진행률 바 + 예상 달성일
- 달성 시 "바로 송금" 옵션 연결

### V4. 출금/송금 전환 (P1)
- Vault → 송금 또는 생활비로 즉시 이동

## 7G. Advance — 소액 가불 (P1)

**"급여일 전에 급할 때, 일한 만큼의 일부를 먼저 — 자금은 Vault 풀에서"**

> 근무 증거(WorkProof)가 있으면 신용 점수 없이도 소액 가불이 가능하다.
> 가불 자금은 Vault 풀에서 나오고, 급여일에 자동 상환된다.
> 한도는 리스크 관리를 위해 **최대 ₩50만**으로 제한.

### 가불 한도 룰
```
가불 한도 = MIN(확인된 근무분 × 20%, ₩50만)

근무 10일 이상 확인  →  최대 ₩30만
근무 15일 이상 확인  →  최대 ₩40만
근무 20일 이상 확인  →  최대 ₩50만 (상한)
이전 달 정상 상환     →  +₩10만 보너스 한도
```

### A1. 가불 가능 금액 계산 (P1)
- 확인된 근태 × 환산시급 × 20%, 상한 ₩50만
- 이전 달 상환 이력에 따라 보너스 한도

### A2. 가불 요청 (P1)
- ₩10만 단위 선택 → 원탭 요청
- 수수료: 정액 ₩1,500/건 (사전 안내 필수)
- 자금 출처: Vault 풀에서 즉시 지급

### A3. 급여일 자동 상환 (P1)
- 급여 입금 시 가불분 + 수수료 자동 차감

### A4. 가불 이력 관리 (P1)
- 이번 달 가불 현황 + 상환 상태

### Vault ↔ Advance 자금 순환
| 흐름 | 설명 |
|---|---|
| Vault → Advance | 예치 풀의 30%가 가불 자금으로 활용 |
| Advance → Vault | 급여일 상환 시 원금 + 수수료가 풀로 회수 |
| 수수료 → 예금 이자 | 가불 수수료 수익이 예치자 이자 재원에 기여 |
| 리스크 관리 | 1인당 최대 ₩50만 × 소수 이용 = 풀 대비 리스크 극소 |

## 8. 인공지능(코파일럿) 설계

### 8.1 목적: “자유대화”가 아니라 “현재 화면을 돕는 설명/문장 생성”
- 각 핵심 화면에 **“이 화면 설명해줘”** 버튼 제공
- 추천 질문 칩(예시)을 함께 제공해 사용자가 무엇을 물어볼지 고민하지 않게 한다.

### 8.2 코파일럿 UX(버튼/칩 예시)
#### (1) 차액 감지 화면
- 버튼: `이 화면 설명해줘`
- 추천 질문 칩:
  - `왜 차액이 생겼어?`
  - `어떤 근거로 그렇게 판단했어?`
  - `지금 내가 할 수 있는 다음 행동은 뭐야?`

#### (2) 송금 차단/보호 화면(SafePay)
- 버튼: `왜 송금이 막혔어?`
- 칩:
  - `24시간 쿨다운이 뭐야?`
  - `얼마부터 고액 확인이야?`
  - `다시 보내려면 어떻게 해?`

#### (3) 신고 준비 화면(Instant Claim)
- 버튼: `제출용 문장 만들어줘`
- 칩:
  - `정중하지만 단호하게 써줘`
  - `짧게 요약해줘`
  - `베트남어로 번역해줘`(P1)

### 8.3 코파일럿이 제공하는 기능(권장 3종, P0)
> 코파일럿은 **설명/번역/문장 생성만** 수행한다.  
> 숫자 계산/판정은 서버 룰 기반 결과를 그대로 사용한다.

1) **차액/정책 차단 이유를 쉽게 설명**(근거 포함)  
2) **사업주 메시지 초안 다국어 생성**(정중+단호, 근거 포함)  
3) **신고 제출용 요약 문장 생성**(반자동 제출 지원)

### 8.4 가드레일(환각/오답 방지 정책) — PRD 고정
- 코파일럿은 **새로운 숫자를 계산하거나 추정하지 않는다.**
- 코파일럿은 서버가 제공한 **facts(사실 데이터)**만 사용해 문장을 만든다.
- facts 예시:
  - 차액 금액, 근무시간 합계, 연장/야간 시간, 해당 WorkProof ID 목록, 수정 사유 목록, SafePay 차단 코드, 허용 목록 상태 등
- 코파일럿이 facts 밖의 내용을 말하려고 하면:
  - “확인할 수 없음/추정 불가”로 응답하고, 필요한 입력을 안내한다.
- 모든 코파일럿 응답에는 화면 하단에 공통 문구를 표시:
  - “이 안내는 화면에 표시된 사실을 바탕으로 작성된 참고 문장입니다.”

### 8.5 RAG 정책: “필수 아님, 필요 시 RAG-lite”
- 대부분 질문은 화면 facts로 해결한다.
- 용어/기능 설명(예: Proof Pack vs Claim Kit, 신고 준비 범위) 정도만 커버하기 위해:
  - 짧은 FAQ/용어집을 **키워드 검색 또는 소규모 임베딩 Top-k**로 제공(RAG-lite)
- “대규모 지식 기반 챗봇”은 P1 이후로 미룬다.

## 9. 데모 전용 기능: 타임 트래블(Time Travel)

> **실사용 기능이 아니라 발표 데모를 위한 장치**이며, 릴리스 빌드에서는 비활성화한다.

### 9.1 목적
- 실제 한 달 사용 흐름(근태 → 월간 요약 → 급여 점검 → 차액 감지 → 문서/메시지/신고 준비)을  
  **15~30초로 압축**해서 보여주기

### 9.2 동작 방식(권장)
- 데모 시작 시 **시드(고정) 데이터**를 주입한다.
  - WorkProof 기록(일자별 출퇴근 + 일부 수정 이력 포함)
  - 급여 이벤트(예상/실입금 입력)
  - 송금 이벤트(성공/실패 1건씩)
  - (P1) Vault 이벤트: 예치/이자 발생 시뮬레이션
  - (P1) Advance 이벤트: 가불 요청/승인/상환
- 사용자는 화면 상단의 슬라이더/재생 버튼으로 `기준일(asOf: YYYY-MM-DD)`을 바꾼다.
- 앱은 기준일 이전 데이터만 반영해 다음 화면을 즉시 갱신한다.
  - 월간 요약, 참고용 추정, 차액 감지, 다음 행동 버튼(Proof Pack/메시지/Claim Kit/신고 준비)

### 9.3 구현 원칙(데모 품질을 위한 기준)
- **“시드 데이터는 고정”**이지만, **집계/계산/판정 로직은 실제 로직**을 사용한다.
- 기준일 변경은 “데모 모드 파라미터”로 서버에 전달한다.
  - 예: 요청 헤더 `X-Demo-AsOf: 2026-03-20`
  - 서버는 해당 날짜 이전의 이벤트만 조회/집계해 응답한다.
- 결과적으로, 슬라이더만 움직여도 실제 사용자처럼 모든 화면이 자연스럽게 이어져야 한다.

### 9.4 데모 모드 정책
- 데모 전용 계정에서만 활성화
- 일반 계정/릴리스 빌드에서는 완전 비활성(설정 숨김)

## 10. 서버 아키텍처(Feature-first) — PRD 구현 가이드

### 10.1 구성 요소(한 대의 서버 + Compose 컨테이너)
- `api-server`(Spring): 외부 공개 API, 모든 비즈니스 규칙
- `pdf-service`(Node): HTML 템플릿 + 브라우저 렌더링으로 PDF 생성(내부 전용)
- `postgres`: 저장소(상태/로그/잡)
- `redis`: 쿨다운/락/보조 큐(권장)
- `s3`: 문서 저장(Private) + presigned URL 다운로드

### 10.2 코드 구조(Feature-first 권장)
```text
com.workproofpay
  ├─ auth
  ├─ workproof
  ├─ wage
  ├─ safepay
  ├─ remittance
  ├─ vault          ← (P1) 예금/목표저축/yield
  ├─ advance        ← (P1) 가불 요청/상환
  ├─ documents
  ├─ jobs
  ├─ notifications
  └─ shared
```

각 기능 모듈 내부는 MVP에서는 단순하게:
- `api`(컨트롤러/요청응답 DTO)
- `service`(유스케이스/정책)
- `repo`(JPA)
- `adapter`(외부 연동)
- `model`(엔티티/도메인 모델)

### 10.3 외부 의존성은 인터페이스로 숨긴다(팀 협업 핵심)
블록체인/파일/PDF/인공지능은 다음 인터페이스 뒤에 둔다.

#### (1) 블록체인 게이트웨이(remittance)
- 팀원은 `TransferUseCase`만 사용
- web3j 사용/nonce/gas는 adapter에서만

```java
public interface TransferUseCase {
  TransferId requestTransfer(RequestTransferCommand cmd); // Idempotency-Key 포함
  TransferDto getTransfer(TransferId id);
}
```

### 10.4 비동기 잡(jobs) 설계(송금/문서 생성 안정화)
- PDF 생성, 송금 전송/확정 확인은 “요청-응답”에서 분리한다.
- `jobs` 테이블로 상태/재시도를 관리한다.

#### 잡 상태
- QUEUED → RUNNING → DONE / FAILED

#### 잡 종류(예시)
- RENDER_PROOF_PACK
- RENDER_CLAIM_KIT
- SUBMIT_TRANSFER
- POLL_TRANSFER_RECEIPT
- RENDER_TRANSFER_RECEIPT

### 10.5 송금이 비동기 처리인 이유(요약)
- 전송은 빠를 수 있지만, **완료 확정은 전송 결과(영수증) 확인이 필요**
- 네트워크 재시도/중복 요청이 흔해 **중복 방지 키와 재시도 정책**이 필수
- nonce 충돌을 피하기 위해 **전송 잡은 직렬화(동시성=1)**가 MVP에 안전

### 10.6 데이터 모델(개요)

- workproof_records + workproof_modifications(수정 이력) + workproof_attachments
- wage_estimates, wage_verifications
- recipients, transfers
- vault_deposits, vault_goals, vault_yield_logs ← (P1)
- advances, advance_repayments ← (P1)
- documents(ProofPack/ClaimKit/Receipt) + document_links(관련 엔티티 연결)
