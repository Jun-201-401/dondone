# 미리받기 회차 정책과 사용자 흐름

## 목적

이 문서는 현재 DonDone 미리받기 기능의 **회차 계산 기준**, **신청 잠금 조건**, **모바일 안내 문구**를 팀 공유용으로 정리한다.

- 대상: backend, mobile, demo 시연 흐름
- 기준 구현:
  - `apps/dondone-backend/.../advance/service/AdvancePolicyEngine.java`
  - `apps/dondone-backend/.../advance/service/AdvanceService.java`
  - `apps/dondone-mobile/.../domain/advance/AdvanceContractState.kt`

## 핵심 요약

- 급여일 기준일은 매월 `25일`이다.
- 미리받기 회차는 `오늘 날짜` 기준으로 계산한다.
- **급여일 당일만** 신규 신청을 막는다.
- 급여일이 지나면 기존 회차를 계속 막아두지 않고 **다음 달 급여 회차**로 자동 전환한다.
- `확인 필요한 기록`은 안내 사유로 표시되지만, 그것만으로 무조건 하드 차단되지는 않는다.

## 회차 계산 기준

### 1. 기준 급여일

- 기본 급여일: 매월 `25일`

### 2. 회차 결정

- 오늘이 `25일 이하`면 현재 달 회차를 사용한다.
- 오늘이 `25일 초과`면 다음 달 회차를 사용한다.
- 다만 최신 근무 기록의 월이 회차 기준보다 더 뒤라면, 그 최신 근무 월을 사용한다.

### 3. 예시

| 오늘 날짜 | 회차(targetMonth) |
| --- | --- |
| 2026-03-20 | 2026-03 |
| 2026-03-25 | 2026-03 |
| 2026-03-26 | 2026-04 |
| 2026-03-28, 최신 근무 월이 2026-05 | 2026-05 |

## 언제 잠기는가

현재 하드 차단은 아래 세 경우다.

### 1. 반영된 근무가 부족한 경우

다음 조건 중 하나라도 만족하면 신청이 잠긴다.

- `reflectedWorkDays == 0`
- `verifiedMinutes == 0`

응답 코드:

- `INSUFFICIENT_VERIFIED_WORK`

모바일 기본 문구:

- `반영된 근무가 더 필요해요`

### 2. 같은 회차에 진행 중인 미리받기가 이미 있는 경우

같은 사용자, 같은 근무지, 같은 회차(`yearMonth`)에 아래 상태 요청이 있으면 신규 신청을 막는다.

- `SUBMITTED`
- `APPROVED`

응답 코드:

- `EXISTING_OUTSTANDING_ADVANCE`

모바일 기본 문구:

- `이미 진행 중인 미리받기가 있어요`

### 3. 급여일 당일인 경우

회차의 상환 예정일이 오늘과 같으면 신규 신청을 막는다.

- 예: 3월 25일에는 3월 회차 신규 신청 불가

응답 코드:

- `ADVANCE_WINDOW_CLOSED_TODAY`

모바일 기본 문구:

- `오늘은 신청이 마감됐어요`

## 언제 다음 달로 넘어가는가

### 1. 급여일 전

- 현재 달 회차로 계산한다.
- 예: 3월 24일 -> 3월 회차

### 2. 급여일 당일

- 현재 달 회차이지만 신규 신청은 막힌다.
- 예: 3월 25일 -> 3월 회차, 신청 불가

### 3. 급여일 다음날부터

- 다음 달 회차로 자동 전환한다.
- 예: 3월 26일 -> 4월 회차

즉, 현재 정책은 `급여일 이후에도 이전 달 회차를 계속 잠가두는 방식`이 아니라,
`다음 달 급여 회차로 자연스럽게 넘기는 방식`이다.

## 확인 필요 기록은 어떻게 처리하는가

다음 조건은 안내 사유로 표시된다.

- `needsReviewRecordCount > 0`
- `pendingMinutes > 0`

응답 코드:

- `PENDING_WORKPROOF_REVIEW`

모바일 기본 문구:

- `확인 필요한 기록이 남아 있어요`

중요:

- 이 사유는 현재 **단독으로는 하드 차단 조건이 아니다.**
- 다만 다른 차단 사유와 함께 보일 수 있다.

## 모바일 시연 기준 문구

### 1. 급여일 당일 마감

- 제목: `오늘은 신청이 마감됐어요`
- 본문:
  - 같은 회차인 경우: `오늘은 신청이 마감됐어요. 다음 회차는 마감 이후 기준으로 확인해 주세요.`
  - 다음 회차가 이미 잡힌 경우: `오늘 회차는 마감됐어요. 내일부터 다음 달 급여 회차 기준으로 확인할 수 있어요.`
- 버튼: `마감 이유 보기`

### 2. 확인 필요한 기록이 남은 경우

- 제목: `확인 후 신청이 가능해요`
- 본문: `확인 필요한 기록이 정리되면 바로 신청 가능 금액을 다시 확인할 수 있어요.`
- 버튼: `기록 확인`

### 3. 다음 달 급여 회차로 넘어간 경우

- 제목: `다음 달 급여 회차 기준으로 확인했어요`
- 본문: `다음 달 급여 회차 기준으로 신청 가능 금액을 불러왔어요.`
- 버튼: `다음 회차 보기`

### 4. 정상 신청 가능 상태

- 제목: `실제 계정 기준으로 신청 가능한 상태예요`
- 본문: `백엔드 응답 기준으로 현재 미리받기 한도를 불러왔어요.`
- 버튼: `미리받기 보기`

## 사용자 흐름

### 흐름 A. 급여일 전

1. 사용자가 금융 홈에서 미리받기 카드를 연다.
2. 현재 달 회차 기준으로 가능 금액을 계산한다.
3. 하드 차단 사유가 없으면 신청 가능 상태로 보인다.

### 흐름 B. 급여일 당일

1. 사용자가 미리받기 화면을 연다.
2. 현재 달 회차는 오늘 마감 상태로 계산된다.
3. 가능 금액은 `0원`으로 보이고 신청 버튼은 열리지 않는다.
4. 사용자에게 `오늘은 신청이 마감됐어요` 문구를 보여준다.

### 흐름 C. 급여일 이후

1. 사용자가 미리받기 화면을 다시 연다.
2. 회차가 다음 달로 자동 전환된다.
3. 다음 달 급여 회차 기준으로 다시 한도를 계산한다.
4. 사용자에게 `다음 달 급여 회차 기준으로 확인했어요` 문구를 보여준다.

## 팀 정렬 포인트

- 더 이상 `급여일까지 3일 이하 무조건 차단` 정책을 기준으로 설명하지 않는다.
- 현재 기준은 `급여일 당일만 차단 + 급여일 경과 후 다음 회차 자동 전환`이다.
- 시연 설명도 이 기준으로 통일한다.

## 시연 시 설명 문장

발표나 데모에서 아래 설명을 그대로 써도 된다.

- `미리받기는 급여일 당일만 마감되고, 급여일이 지나면 다음 달 급여 회차 기준으로 자동 전환됩니다.`
- `확인 필요한 출퇴근 기록이 남아 있으면 그 사유를 먼저 보여주고, 정리되면 바로 다시 신청 가능 금액을 확인할 수 있습니다.`
- `현재 금액은 반영된 근무 기록 기준의 데모 시뮬레이션이며 실제 금융 서비스 제공을 의미하지 않습니다.`
