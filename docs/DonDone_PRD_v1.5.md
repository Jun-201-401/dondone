# PRD v1.5 — DonDone
> **"당신의 돈(Don), 처음부터 끝(Done)까지 던던하게."**
> **DonDone은 외국인 근로자를 위한 급여 지갑이다.**
> 근로자에게는 일한 기록이 미리받기, 급여 확인, 증빙으로 이어지고,
> 회사에게는 같은 기록이 선지급과 정산 리스크를 줄이는 운영 근거가 된다.

## 0. 처음 보는 사람을 위한 설명

### 0.0 브랜드 표기 원칙(v1.5)
- 서비스명 표기는 **DonDone**을 기본으로 사용한다.
- 발표/데모 화면의 핵심 슬로건은 아래 문구를 고정 사용한다.
  - **"당신의 돈(Don), 처음부터 끝(Done)까지 던던하게."**

### 0.1 한 문장 요약
> **"일한 기록이 지금 쓸 수 있는 돈이 되고, 월급이 이상하면 같은 기록으로 바로 설명하고 증빙할 수 있는 앱."**

발표/Q&A에서는 아래 한 줄로도 설명할 수 있다.
- **근로자에게는 급여 지갑, 회사에게는 선지급·정산 리스크를 줄이는 운영 콘솔**

### 0.2 이 제품이 해결하려는 핵심 문제
근로자와 사업주는 같은 근무 데이터를 두고 서로 다른 문제를 겪는다.

근로자 관점:
- **급여일 전 현금 부족**: 이미 일했지만 급여일 전까지는 쓸 수 있는 돈이 없어 생활비가 막힐 수 있다.
- **가족 송금의 번거로움**: 소액이라도 자주 보내야 하는데, 수수료와 속도, 영수증 관리가 모두 부담이 된다.
- **월급 확인의 어려움**: 연장·야간·공제 때문에 "내가 받아야 할 돈"과 "실제로 받은 돈"의 차이를 이해하기 어렵다.
- **문제 발생 시 근거 부족**: 월급이 이상하거나 누락이 생겨도 "내가 일했다"는 기록이 약하면 설명과 증빙이 어렵다.
- **남는 돈 관리의 부재**: 다음 송금이나 생활비 전환 전까지 돈을 따로 보관할 선택지가 흩어져 있다.

사업주 관점:
- **운영 관리의 부담**: 출퇴근 누락, 수정 요청, 의사소통 오류, 월말 정산 이슈를 미리 파악하기 어렵다.
- **선지급 판단의 부담**: 누가 안정적으로 정산될 수 있는지 근거 없이 판단하면 운영 리스크가 커진다.
- **분쟁 대응의 부담**: 근무 기록과 정산 근거가 약하면 차액 설명, 확인 요청 대응, 분쟁 정리가 모두 어려워진다.
- **복지 제공의 부담**: 근로자에게 선지급 같은 복지를 주고 싶어도, 회사 현금흐름 부담을 크게 늘리지 않으면서 운영할 구조를 설명하기 어렵다.

### 0.3 우리가 만드는 해법의 핵심
- DonDone은 단순한 출퇴근 앱이 아니라, **근무 기록을 돈의 행동으로 바꾸는 급여 지갑**이다.
- 근로자에게는 같은 기록이 **미리받기, 월급 확인, 설명 요청과 증빙**으로 이어진다.
- 회사에게는 같은 기록이 **선지급 판단, 정산 미리보기, 분쟁 대응**에 쓰이는 운영 근거가 된다.
- 회사 입장에서는 이 구조가 **현금흐름 부담을 크게 늘리지 않으면서 선지급 복지를 설명할 수 있는 틀**로 보여야 한다.
- 사용자는 나중에 월급을 검산하려고 기록하는 것이 아니라, **오늘 기록이 지금 쓸 수 있는 돈을 늘려주기 때문에** 기록해야 한다.
- 따라서 홈은 기록 목록보다 먼저 **"지금 쓸 수 있는 돈"과 "다음 행동"**을 보여주는 화면이어야 한다.
- 미리받기와 송금은 전면 가치이고, 월급 확인과 증빙은 문제가 생겼을 때 이어지는 보호 기능이다.
- DonDone은 **근로자 중심 제품**을 유지하되, P0에서는 회사용 운영 콘솔과 검증 요약 카드로 리스크 절감 축도 함께 보여준다.
- 보관/이자는 테스트넷/데모 범위에서만 보여 주되, 투자나 수익 보장처럼 보이지 않게 표현한다.

## 1. 문제 정의

### 1.1 돈의 흐름에서 발생하는 문제(전체 구간)
```text
일한다 ──→ 근무가 반영된다 ──→ 급여 전 현금 필요 ──→ 급여 받는다 ──→ 생활비/가족송금 ──→ 남는 돈 보관
  │               │                      │                    │                     │
  ▼               ▼                      ▼                    ▼                     ▼
기록 약함         반영 상태 불투명         미리 쓸 돈 부족        누락/체불 확인 어려움   보관/이자/영수증 관리 분산
```

DonDone이 보는 핵심 문제는 **근무 기록이 돈의 흐름과 분리되어 있다는 점**이다.

근로자 관점:
- 이미 일했는데도 급여일 전에는 쓸 수 있는 돈이 부족하다.
- 가족에게 자주 보내야 하지만 송금은 번거롭고 기록 관리도 어렵다.
- 월급이 맞는지 확인하기 어렵고, 차이가 생겨도 무엇이 빠졌는지 알기 어렵다.
- 문제가 생겼을 때 설명과 증빙에 쓸 근무 근거가 부족하다.

회사 관점:
- 누락 기록, 수정 요청, 위치 불일치가 월말에 한꺼번에 쌓인다.
- 선지급 가능 여부와 정산 이슈를 미리 보지 못하면 운영 부담과 분쟁 위험이 커진다.
- 정산 차이와 근무 이력을 설명할 기준이 약하면 확인 요청과 대응 비용이 커진다.

### 1.2 기존 대안의 한계
| 기존 방식 | 되는 것 | 안 되는 것 |
|---|---|---|
| 근태관리 앱 | 출퇴근 기록 | 돈의 흐름(미리받기/월급 확인/송금)과 직접 연결되지 않음 |
| 급여 계산기 | 월급 계산 | 근무 근거와 연결이 약하고, 실제 행동(문서/공유/다음 금융 액션)으로 이어지지 않음 |
| 송금 앱/은행 | 송금 | 내가 일한 기록과 월급 확인, 이후의 근거 관리와 연결되지 않음 |
| 개인 지갑 앱 | 코인 전송/보관 | 외국인 사용자 관점의 안전정책·생활형 UI·근무 기반 신뢰가 부족함 |
| 가불/대출 서비스 | 현금 확보 | 근무 반영 데이터와 급여 보호 기능이 한곳에 있지 않음 |
| 사업장 관리 도구 | 출퇴근/인력 운영 관리 | 선지급 판단, 근로자 보호 흐름, 정산 근거 관리가 분리되어 있음 |

## 2. 타깃 사용자

### 2.1 메인 타깃(MVP 핵심 사용자)
- 제조/건설/농축산/서비스 등 현장 중심 업종의 외국인 근로자
- 시급/일급/월급 혼재, 연장/야간 가능성 존재
- 한국어 급여 문서 이해가 어렵고, 월급 확인이 번거로운 사용자
- 매월 소액 송금이 잦고, 급여일 전 현금흐름 압박을 자주 느끼는 사용자
- 근무 기록이 금융 행동으로 바로 연결될 때 가장 큰 가치를 느끼는 사용자

### 2.2 보조 타깃(P0 지원 사용자)
- 외국인 근로자를 고용하고 있는 사업장 관리자 또는 실무 담당자
- 출퇴근 누락, 수정 요청, 정산 이슈, 선지급 가능 대상 확인이 필요한 사용자
- 채용 점수보다 **운영 리스크 절감과 정산 가시성**에 관심이 있는 사용자
- 근로자 보호 흐름을 해치지 않는 범위에서 선지급과 정산 운영을 더 안정적으로 보고 싶은 사용자
- 직원 만족과 복지를 높이면서도 회사 현금흐름 부담을 크게 늘리지 않는 방식을 찾는 사용자

### 2.3 메인 타깃 JTBD
1) **일한 만큼 지금 할 수 있는 금융 행동이 늘어나는 걸 보고 싶다.**
2) **급하면, 반영된 근무 기록을 바탕으로 일부를 먼저 받고 싶다.**
3) **가족에게 빠르고 간단하게 보내고 싶다.**
4) **월급이 이상한지(누락/지연/공제 과다)를 쉽게 알고 싶다.**
5) **문제가 생기면 같은 기록으로 바로 설명하고 증빙하고 싶다.**
6) **남는 돈을 따로 보관하고, 필요하면 다시 송금이나 생활비로 전환하고 싶다.**

### 2.4 보조 타깃 목표(P0 지원축)
1) **출퇴근 누락, 수정 요청, 위치 불일치를 한 화면에서 빨리 확인하고 싶다.**
2) **선지급 가능 대상과 정산 이슈 가능성을 미리 보고 운영 리스크를 줄이고 싶다.**
3) **자동 점수보다 설명 가능한 요약 카드와 근거 자료로 확인하고 싶다.**
4) **회사 현금흐름 부담을 크게 늘리지 않으면서도 근로자에게 선지급 복지를 제공하고 싶다.**

## 3. 제품 목표

> 중심은 근로자용 급여 지갑이다. 다만 P0에서는 미리받기와 정산 기능을 더 설득력 있게 보여주기 위해, 회사용 운영 리스크 절감 축도 함께 포함한다.

### 3.1 Demo MVP에서 반드시 보여줄 것(P0)
근로자 전면 가치:
- **급여 지갑 홈(Money Home)**: 이번 달 내 돈 상태 + 다음 행동 추천 + 빠른 금융 CTA
- **미리받기(데모 시뮬레이션)**: 반영된 근무 기반 가능 금액 + 신청 + 상환 예정일
- **테스트넷 송금**: 수신자 허용 목록 + 전송 + 상태 추적 + 영수증
- **근무 기록(근거화)**: 출근/퇴근 기록 + 수정 사유 필수 + 수정 이력 자동 저장

근로자 보호 흐름:
- **급여 확인(보호 흐름)**: 이번 달 반영 근무 기준 예상 금액 확인 + 실제 지급 결과 확인 + 차이 근거 제시
- **문제 발생 시 액션(정산 문제 해결)**: 회사와 1차 확인 / 설명 요청 문장 / 증빙 리포트 / 근거 자료 묶음 / 외부 신고 준비
- **문서(PDF)**: 증빙 리포트, 근거 자료 묶음, 송금 영수증을 저장하고 앱에서 공유/다운로드

회사 지원축(P0 지원 기능):
- **고용주 운영 기능**: 회사 확인 코드 + 운영 콘솔 + 선지급 가능 대상 확인 + Verified Worker Summary

신뢰/데모 장치:
- **보관/이자(데모 시뮬레이션)**: 스테이블코인 보관 잔액 + 예상 이자 + 송금 연결
- **무결성 증빙**: 근로계약서/월별 근무 기록/Proof Pack 해시와 확인 시점 타임스탬프를 남겨 나중에 조작되지 않았음을 설명

### 3.2 성공 기준(데모 관점)
- 앱을 켰을 때 5초 안에 "근무 기록 앱"보다 **"급여 지갑/돈 앱"**이라는 인상이 들어야 한다.
- 30초 안에 **일한 만큼 쌓인다 -> 일부 먼저 쓴다/보낸다 -> 월급을 확인한다 -> 문제 시 보호된다** 흐름이 이해돼야 한다.
- 3분 데모에서 "한 달 여정"을 **15~30초로 압축해서 보여주는 데모 장치(Time Travel)**로 스토리를 완주한다.
- "이번 달 쌓인 돈 확인 -> 실제 받은 돈 확인 -> 문제 시 회사와 1차 확인 -> 근거 확인/문서 생성 -> 외부 신고 준비"까지 **끊기지 않는 보호 흐름**을 구현한다.
- 고용주 관점에서도 **"회사 확인 코드와 운영 콘솔을 바탕으로 선지급·정산 리스크를 줄이고, 회사 현금흐름 부담을 크게 늘리지 않으면서 선지급 복지를 운영할 수 있는 도구"**라는 설명이 30초 안에 전달돼야 한다.

## 4. 범위(P0/P1) 확정표

> 원칙: P0는 **금융 중심으로 확실히 되는 데모**, P1은 **정교화/실서비스 확장**이다.  
> P0의 중심은 **근로자 전면 가치와 보호 흐름**이고, 고용주 기능은 이를 뒷받침하는 **운영 리스크 절감 지원축**으로 포함한다.  
> 미리받기/보관은 P0에서 **데모 시뮬레이션**으로 보여주고, 실제 정책/실거래 연결은 P1로 미룬다.

| 영역 | P0(6주 Demo MVP) | P1(후속) |
|---|---|---|
| 금융 홈 | 지금 쓸 수 있는 돈 중심의 홈, 다음 행동 추천, 빠른 이동 | 개인화 추천, 리마인드/알림 고도화 |
| 근무 기록 | 출근/퇴근 원탭, 수정 사유 필수, 첨부 선택, 수정 이력 자동 포함 | 지오펜스 기반 자동 알림, 다중 근무지 |
| 미리받기 | **P0 전면 가치**: 가능 금액, 신청, 상환 예정일, 근거 설명 | 실제 정책/파트너 연동, 한도 고도화 |
| 고용주 기능 | **P0 지원축**: 운영 콘솔, 선지급 가능 대상 확인, Verified Worker Summary, 회사 확인 코드 | 고용주 포털 고도화, Work Passport 공유/검증, 정산 확인 자동화 |
| 급여 점검 | **P0 보호 흐름**: 이번 달 반영 근무 기준 예상 금액 확인 + 실제 지급 결과 확인 + 이상 징후 탐지(근거 포함), 공제 미입력 배지/임계값 완화 | 급여명세서/계약서 자동 추출 고도화 |
| 송금 | 테스트넷 전송, 상태 추적, 영수증, SafePay | 정기 송금/자동 송금, 한도 고도화 |
| 보관/이자 | **P0 데모 장치**: 잔액, 예상 이자, 송금 전환 | 메인넷 연동, 실 yield 파트너십 |
| 문서(PDF) | **P0 보호/신뢰 장치**: 증빙 리포트/근거 자료 묶음/송금 영수증 PDF(템플릿 v1) | 문서 템플릿 확장, 다국어 템플릿 |
| 정산 문제 해결 | **P0 보호 흐름**: 문제 발생 시 회사와 1차 확인 + 설명 요청 문장 + 증빙 리포트 + 근거 자료 묶음 + 외부 신고 준비(Instant Claim v0) | 기관 제출 자동화 또는 제휴 연동 |
| 인공지능(코파일럿) | 화면 설명/번역/요약 문장 생성 3종 + facts 기반 가드레일 | RAG-lite 고도화, 언어 확장 |
| RAG | **P0 비핵심**: 기본은 화면 facts | FAQ/용어집 기반 RAG-lite |

| 정산 문제 해결 | 확인이 필요한 차이가 있을 때 회사와의 1차 확인, 설명 요청, 증빙 리포트, 근거 자료 묶음, 외부 신고 준비까지 이어지는 보호 흐름 |
| 증빙 리포트 | 근무 로그, 계산 근거, 차액 리포트를 한 파일(PDF)로 묶은 문서. 내부명은 Proof Pack (`PROOF_PACK`) |
| 근거 자료 묶음 | 제출/상담에 필요한 요약, 근거, 첨부를 함께 묶은 문서. 내부명은 Claim Kit (`CLAIM_KIT`) |
| 신고 준비(Instant Claim v0) | 정산 문제 해결 흐름의 마지막 단계에서 제출을 더 쉽게 만드는 반자동 지원 기능 |
| 보관/이자(Vault) | 남는 스테이블코인을 따로 보관하고 예상 이자를 확인하는 기능. P0에서는 데모 시뮬레이션 |
| 허용 목록 | 등록된 수신자에게만 송금할 수 있게 제한하는 목록 |
| 중복 방지 키(Idempotency Key) | 같은 요청이 여러 번 와도 한 번만 처리되게 하는 요청 키. 송금과 문서 생성에 필수 |
| 전송 해시(Tx Hash) | 블록체인 전송의 고유 ID. 상태 추적과 영수증에 사용 |
| 무결성 해시(Integrity Hash) | 근로계약서, 월별 근무기록, Proof Pack 같은 핵심 문서의 해시를 남겨 이후 위변조 여부를 설명하는 값 |

## 6. 핵심 사용자 플로우(특히 ‘돈의 흐름’ 중심)

### 6.1 최초 1회: 기본 설정
1) 근무지 등록(이름/주소)
2) 급여 정보 등록: **기본급 1줄 + 단위 선택(시급/일급/월급)**
3) 송금 수신자 등록(허용 목록)
4) 홈에서 `지금 쓸 수 있는 돈`과 다음 행동 확인
5) 필요 시 미리받기 / 월급 확인 / 가족 송금 / 보관으로 이동
6) 회사와 연결하는 경우 회사 확인 코드 등록

### 6.2 매일: 출근/퇴근(근무 증거 생성)
- 출근 탭 -> 기록 생성
- 퇴근 탭 -> 기록 확정
- (필요 시) 근무 시간 수정 -> **수정 사유 필수 + 추가 증거 선택 첨부**
- 앱은 기록 결과를 **"오늘 근무가 반영되면 지금 어떤 금융 행동이 가능해지는지"**와 연결해 보여준다.

### 6.3 급여일 전: 미리받기
1) 반영된 근무일/시간 확인
2) 지금 쓸 수 있는 돈과 미리받기 가능 금액 확인
3) 신청 금액 선택 -> 확인
4) 결과:
   - 가능: 신청 완료 + 상환 예정일 확인
   - 보완 필요: 미반영/확인 필요 기록 안내

### 6.4 매월: 급여 점검
1) 월간 요약과 **이번 달 반영 근무 기준 예상 금액** 확인
2) 참고용 급여 추정 확인("참고/추정" 표기)
3) 급여일 이후 **실제로 받은 돈 확인**
   - 근로자: 실수령 입금액 직접 입력 또는 확인 메모 등록
   - 회사(연결된 경우): 운영 콘솔에서 확인 요청과 근거 요약을 보고 설명/정정 필요 여부 검토
4) 결과:
   - 정상: "차이 없음(또는 허용 범위)"
   - 이상: "확인 필요한 차이" + "근거 제시" + "회사와 1차 확인"

### 6.5 확인이 필요한 차이 발생 후 액션(정산 문제 해결)
확인이 필요한 차이 화면에서 한 번에 제공:
1) **회사와 직접 확인하기**: 설명 요청 문장 / 정정 요청 문장 / 확인 상태 표시
2) **증빙 리포트 만들기**
3) **근거 자료 묶음 만들기**
4) **외부 신고 준비(Instant Claim v0)**: 회사와의 1차 확인으로 해결되지 않았을 때 제출 경로 안내 + 자동 요약 문장 + 체크리스트 + 파일 공유 + 바로가기

> 연결된 회사가 있는 경우에는 `바로 신고`보다 **회사와의 1차 확인 -> 근거 정리 -> 외부 신고 준비** 순서를 기본 흐름으로 둔다.

> 핵심 UX 목표: 메인 가치가 금융이어도, 문제가 생기면 **근거 -> 문서 -> 다음 행동**이 같은 흐름 안에서 이어져야 한다.

### 6.6 송금(테스트넷)
1) 수신자 선택(허용 목록)
2) 금액 입력 -> 확인
3) 전송 후 상태 추적(SUBMITTED -> CONFIRMED/FAILED)
4) 영수증 PDF 생성 및 공유

### 6.7 보관/이자(데모 시뮬레이션)
1) 보관 가능한 잔액 확인
2) 예상 이자 확인
3) 필요 시 송금 또는 생활비로 전환

### 6.8 고용주: 운영 콘솔과 요약 카드 확인
1) 회사 확인 코드로 사업장 연결
2) 오늘 출퇴근 현황, 누락/수정 요청, 위치 불일치 확인
3) 월말 정산 미리보기와 급여 차액 가능성 확인
4) 근로자 확인 요청 상태와 근거 요약을 함께 보고 설명/정정 필요 건 분류
5) 선지급 가능 대상과 Verified Worker Summary 확인

> 원칙: 고용주 기능은 근로자를 점수로 줄 세우는 도구가 아니라, **선지급과 정산 운영 리스크를 줄이는 보조 기능**으로 설계한다.

## 7. 기능 요구사항(상세)

## 7A. 금융 홈(Money Home)

### 핵심 원칙
- 첫 인상은 **급여 지갑/핀테크**여야 한다.
- 홈은 "기록 화면"이 아니라 **"이번 달 내 돈 상태"와 "지금 쓸 수 있는 돈"을 보여주는 화면**이어야 한다.
- 가장 먼저 눈에 들어오는 값은 `예상 급여`보다 **`지금 쓸 수 있는 돈`**이어야 한다.
- 사용자가 지금 해야 할 행동을 **1개 CTA**로 강하게 제안

### H1. 메인 Hero 카드(P0)
- 제목은 **"이번 달 내 돈"** 기준으로 구성
- 필수 정보:
  - 지금 미리받기 가능 금액
  - 지금 보낼 수 있는 금액
  - 예상 급여
  - 실제 입금 상태
- 우측 또는 하단에 다음 행동 CTA 1개 노출

### H2. 상태 기반 추천 행동(P0)
- 급여 전: `미리받기 보기`
- 급여일 전후: `이번 달 쌓인 돈/실제 받은 돈 확인`
- 차액 발생 시: `정산 문제 해결하기`
- 급여 후: `송금하기`
- 여유 자금 존재 시: `보관하고 이자 보기`

### H3. 빠른 금융 액션(P0)
- 미리받기
- 가족 송금
- 보관/이자
- 월급 확인

### H4. 오늘 근무 카드의 역할 조정(P0)
- 홈에 남기되, 핵심 카드가 아니라 보조 카드로 둔다.
- 문구는 기록 그 자체보다 금융 연결을 강조한다.
  - 예: "오늘 근무가 반영되면 미리받기 한도가 늘 수 있어요"
  - 예: "오늘 퇴근을 찍으면 지금 쓸 수 있는 돈이 늘어날 수 있어요"
  - 예: "퇴근 기록이 없으면 이번 달 반영이 지연될 수 있어요"

### H5. 상태 표현(P0)
- loading / empty / error / success 상태를 명시적으로 구분한다.
- 금액이 없는 경우에도 `왜 없는지`와 `다음 행동`을 함께 보여준다.

## 7B. 근무 증거(WorkProof)

### W1. 근무지 등록(P0)
- 이름, 주소, 지도 선택

### W2. 급여 정보 입력(P0, 단순 UI)
- 입력 UI는 **"기본급 1줄 + 단위 선택(원/시간, 원/일, 원/월)"**로 통일
- 단위별 조건 입력(1개만 추가 노출)
  - 일급: 하루 기준 근무시간(기본 8시간, 수정 가능)
  - 월급: 월 기준 근무시간(기본값 제공, 수정 가능)
- 내부 로직은 **시간당 기준으로 정규화(환산 시급)**하여 동일 엔진 재사용

### W3. 출퇴근 원탭 기록(P0)
- 출근/퇴근 시점에 위치 스냅샷 1회 저장(백그라운드 추적 없음)
- 기기 시간 + 서버 수신 시간 함께 기록(감사 로그 기반)

### W4. 근무 시간 수정(‘증빙 강화 UX’로 설계) (P0)
**목표:** "수정 가능 = 조작"이 아니라 **"수정 가능 = 설명 가능한 기록"**으로 신뢰 강화

#### W4-1 수정 사유 선택(필수)
- 사용자가 시간을 수정할 때 **수정 사유 선택이 필수**
- 예시(초기 제공):
  - 출근/퇴근 탭을 늦게 눌렀어요
  - 연장근무가 있었어요
  - 휴게시간이 달랐어요
  - 기록이 누락됐어요
  - 기타(직접 입력)

#### W4-2 추가 증거 첨부(선택)
- 사진(예: 작업지시서, 근무표, 현장 사진 등) / 메모 첨부 가능
- 첨부는 선택이며, "첨부 없음"도 명확히 기록

#### W4-3 수정 이력(감사 로그) 자동 저장
- 수정 시 아래 정보를 **자동으로 기록**
  - 누가: 사용자 ID(또는 계정)
  - 언제: 수정 시각(서버 시간)
  - 무엇을: 변경 전/후 시간(또는 변경된 필드)
  - 왜: 수정 사유(선택값 + 기타 메모)
  - 증거 유무: 첨부 개수/종류(사진/메모) 및 파일 참조
- 수정 이력은 **삭제 불가(감사 목적)**, 단 사용자는 "추가 메모"는 보완 가능(기록으로 남김)

#### W4-4 문서 자동 반영 명세
- 증빙 리포트/근거 자료 묶음에는 다음이 자동 포함되어야 함:
  - 수정된 기록 표시(배지)
  - 수정 이력 표(변경 전/후, 사유, 첨부 유무, 시각)
  - 첨부가 있는 경우: 첨부 목록(파일명/업로드 시각/해시 또는 식별값)

### W5. 월간 요약(P0)
- 월간 근무일, 총 근무시간, 연장/야간 시간 요약
- "수정된 기록 수" 요약 포함
- "반영 완료 / 확인 필요 / 미반영" 상태를 함께 표시해 금융 연결성을 높인다.

### W6. 금융 연결 상태(P0)
- 근무 화면은 단순 일지보다 **"금융 반영 현황"**으로 보이도록 구성
- 필수 표시:
  - 이번 달 반영 근무일
  - 확인 필요한 기록 수
  - 미리받기/월급 확인에 사용할 수 있는 기준일

### W7. WorkProof Integrity 판정(P0)
- WorkProof는 단순 기록 저장이 아니라, **"이 근무를 금융 판단에 써도 되는가"**를 가르는 근거로 사용한다.
- 위치 스냅샷(GPS), 기기 시간, 서버 수신 시간, 수정 사유, 감사 로그는
  **"정말 일했는가"**를 확인하기 위한 기술 요소로 사용한다.
- P0에서는 아래 상태를 구분한다.
  - 기록된 근무
  - 반영 근무
  - 검증 완료 근무
  - 선지급 가능 근무
- 확인에 쓰는 근거 예시:
  - 출근/퇴근 위치 스냅샷
  - 기기 시간 / 서버 수신 시간
  - 수정 횟수 / 수정 사유 / 첨부 유무
  - 기기 일관성 및 데모용 위험 플래그
- 내부 계산 예시:
  - `verified_hours`
  - `pending_hours`
  - `workproof_risk_flags`
- P0에서는 복잡한 부정 탐지 모델 대신, **설명 가능한 규칙과 감사 로그**로 근무 신뢰를 판단한다.

## 7C. 미리받기(Advance, 데모 시뮬레이션)

### 핵심 원칙
- 미리받기는 P0에서 **사용자가 가장 먼저 체감하는 가치**로 보이되, 실제 상용 가불처럼 과장하지 않는다.
- 이 기능이 있어야 사용자가 **"기록하면 바로 내 돈이 열린다"**는 이유를 체감할 수 있다.
- 사용자는 **"왜 이 금액이 가능한지"**, **"왜 막혔는지"**를 이해할 수 있어야 한다.
- DonDone의 미리받기는 범용 대안신용평가가 아니라, **검증된 근무와 회수 가능성을 함께 보는 서비스 내부 정책**으로 설명한다.

### A0. 정책 구조(P0)
- P0 미리받기 정책은 아래 세 층으로 구성한다.
  - **근무 검증**: 정말 일했는가
  - **상환 가능성 확인**: 급여가 들어오고 회수될 가능성이 높은가
  - **차단 규칙**: 위험이 큰 상황은 먼저 막는가
- 필요 시 회사 확인 코드, 재직 확인, 최근 정산 이슈 여부 같은 **고용주 확인 신호**를 지원 근거로 사용한다.
- 즉, GPS 같은 기술은 **근무 검증**에 쓰고, 채무 불이행 위험은 **급여 흐름 확인과 차단 규칙**으로 함께 관리한다.
- P0에서는 실제 금융 연동 없이도, 시드 데이터와 `asOf` 기준으로 위 상태가 자연스럽게 변해야 한다.

### A0-1. 정책 배경(외부 참고)
- CFPB의 2024년 7월 18일 Data Spotlight는 **급여 흐름 확인과 보수적인 회수 구조가 손실 관리에 중요할 수 있음**을 보여주는 참고 자료다.
- DonDone은 이 점을 반영해, 위치정보만으로 판단하지 않고 **검증된 근무 기록 + 급여 흐름 확인 + 차단 규칙**을 함께 사용하는 방향으로 설계한다.

### A1. 가능 금액 계산(P0, 시뮬레이션)
- 선지급 가능 근무액 = `verified_hours × normalized_hourly_wage`
- 기본 한도 = `선지급 가능 근무액 × tier_ratio`
- 최종 한도 = `min(기본 한도, tier_cap, payday_cap)`
- 데모 기준 상한은 **최대 ₩50만**
- 근무 반영 상태와 상환 가능성에 따라 금액이 달라짐을 명시
- P0 예시 tier:
  - D: 0원
  - C: 10%, 최대 5만원
  - B: 20%, 최대 15만원
  - A: 30%, 최대 30만원

### A2. 신청 UX(P0, 시뮬레이션)
- 추천 금액 칩 또는 직접 선택
- 예상 수령액 / 수수료 / 상환 예정일 표시
- 막힌 이유가 있으면 바로 설명
- 확인 후 신청

### A3. 근거 설명(P0)
- 아래 요소를 함께 표시:
  - 반영 근무일 수
  - 검증 완료 근무시간(`verified_hours`)
  - 확인 필요 근무시간(`pending_hours`)
  - 확인 필요 기록 수
  - 적용된 한도 단계(`repayment_tier`) 또는 차단 사유
  - 다음 구간까지 얼마나 더 필요한지

### A3-1. Repayment Confidence 판단값(P0)
- P0에서는 아래 신호를 바탕으로 **"지금 열어도 회수될 가능성이 높은가"**를 판단한다.
  - 급여 수령 계좌 확인 여부
  - 최근 급여유입 확인 여부 및 관측 횟수
  - 회사 확인 코드 또는 재직 확인 여부
  - 급여일까지 남은 일수
  - 계약 종료 예정일 또는 마지막 급여 주기 진입 여부
  - 기존 미상환 여부
  - 계정/기기/근무 기록에서 관찰된 고위험 플래그
- 내부 판단 결과 예시:
  - `repayment_tier`
  - `max_advance_limit`
  - `block_reason_codes`
- 사용자에게는 복잡한 점수 대신 **가능 금액, 근거, 차단 사유**를 중심으로 보여준다.
- P0에서는 관리자 전용 심사 화면을 두지 않고, **데모 시드 데이터 + 타임트래블 상태 변화**로 위 값을 재현한다.

### A3-2. Hard Stop Rules(P0 예시 정책)
- 아래 경우에는 점수 계산보다 먼저 신규 미리받기를 제한할 수 있다.
  - 급여 수령 계좌 미연결
  - 최근 급여유입 미확인
  - 회사 확인이 필요한 구간인데 재직 확인 실패
  - 기존 미상환 잔액 존재
  - 고위험 부정/근무 플래그 존재
  - 마지막 급여 주기 진입
- 이 규칙의 목적은 점수를 높게 보이게 하는 것이 아니라, **채무 불이행 위험이 큰 상황을 먼저 막는 것**이다.
- 계약 종료 예정일이 **30일 이내**로 들어오면 경고 또는 한도 축소 구간으로 본다.
- 계약 종료 예정일이 **14일 이내**이거나 마지막 급여 주기 진입 시에는 신규 신청 차단을 기본 정책 예시로 둔다.
- 급여일까지 **3일 이하** 남은 경우에는 신규 한도를 추가로 축소하거나 차단할 수 있다.

### A4. 이력 관리(P0)
- 이번 달 신청 이력
- 누적 사용 금액
- 상환 예정일

### A5. 실제 상용화(P1)
- 실제 정책/파트너 연동
- 공과금/통신/월세 등 보조 신호 활용 검토
- 오픈뱅킹/마이데이터 기반 급여유입 및 상환경로 확인
- 리스크 정책, 사용자 등급, 상환 정책 고도화

## 7D. 급여 점검(Wage Shield) — ‘정답 계산기’가 아니라 ‘이상 탐지’

### 핵심 원칙(문구/디스클레이머)
- 급여 계산 결과는 **"참고용 추정"**이며, 실제 지급/공제는 근로계약·법규·급여명세서에 따라 달라질 수 있다.
- DonDone의 역할은 **월급을 대신 확정하는 것**이 아니라, **이상 징후를 빠르게 찾고 근거를 보여주는 것**이다.
- Wage Shield는 **근로자가 먼저 이번 달 반영 근무와 실제 받은 돈을 확인하는 보호 레이어**다.
- 회사는 연결된 경우에만, 근로자의 확인 요청 이후 운영 콘솔에서 설명/정정에 참여하는 보조 축으로 동작한다.
- 앱은 법률 자문이 아니다.

### WS1. 참고용 급여 추정(P0)
- 입력된 급여 정보(시급/일급/월급 환산) + WorkProof 기반 시간 집계로 추정
- 중복 계산 방지(프리미엄은 추가분만)
  - 기본: 시급 × 총 근로시간 × 1.0
  - 연장 프리미엄: 시급 × 연장시간 × 0.5
  - 야간 프리미엄: 시급 × 야간시간 × 0.5
- 화면에는 항상 근거를 함께 표시:
  - 시간/구간/WorkProof ID
  - 수정 사유 유무(수정 배지)

### WS2. 실제 지급 결과 반영(P0)
- 기본값은 근로자의 "통장 입금액(실수령)" 직접 입력이다.
- 회사가 연결된 경우 운영 콘솔에서 해당 월의 확인 요청 상태, 근거 요약, 정산 설명 메모를 함께 본다.
- 화면에는 실제 지급 결과의 **반영 상태**를 표시한다.
  - 예: 근로자 직접 입력
  - 예: 회사와 1차 확인 진행 중
  - 예: 회사와 추가 확인 필요
- 공제 항목을 입력하지 않은 경우:
  - **"공제 미반영" 배지** 표시
  - 알림 임계값을 완화(불필요 경보 최소화)

### WS3. 차이 확인(이상 징후 탐지) (P0)
- 임계값(기본값)
  - 차액 ≥ 30,000원 또는 예상액의 2%
  - 공제 미반영이면 50,000원 또는 3%로 완화
- 차이 확인 화면에서는 `차이가 있다`에서 끝나지 않고, **가능한 원인과 근거**를 함께 제시한다.
  - 예: 연장시간 합계, 야간시간 합계, 수정 기록 포함 여부
  - 예: 근로자 확인 요청 이후 회사와 1차 확인이 필요한 상태 여부

### WS4. 근거 중심 출력(P0)
- "총 차액"만 보여주지 않고, **항목별 근거**를 제공:
  - 연장/야간 합계 시간
  - 해당 WorkProof 목록(링크)
  - 수정 사유/첨부 유무 표시
  - 실제 지급 결과의 반영 상태와 회사 확인 진행 상태

### WS5. 문서 파싱 고도화(P1)
- 급여명세서/계약서 자동 추출(후속)
- MVP 단계에서는 수동 입력이 기본이며, 파싱은 "있으면 편한 기능"으로 제한

## 7E. 송금(Remittance, 테스트넷)

> 송금은 부가 기능이 아니라, 사용자가 DonDone의 가치를 바로 체감하는 두 번째 금융 행동이다.

### R1. 수신자 허용 목록(P0)
- 별칭/관계/사진(선택) 포함
- 허용 목록 외 송금 불가
- "자주 보내는 사람" 표시 지원 권장

### R2. 송금 실행(P0)
- 테스트넷에서 스테이블코인(예: USDC 또는 모의 토큰) 전송
- 전송 전 반드시 사용자 확인(확인 화면)
- 송금 전 화면에는 아래를 명시:
  - 허용 목록 수신자
  - 데모 송금(실자금 이동 없음)
  - 영수증 저장 예정

### R3. 상태 추적(P0)
- 상태: SUBMITTED -> CONFIRMED / FAILED
- 실패 시 재시도 안내(정책 기반)
- 송금 상세 화면에서 상태와 전송 해시를 즉시 다시 열 수 있어야 함

### R4. 영수증(P0)
- 전송 해시(Tx Hash) 포함
- 영수증 PDF 자동 생성(문서 모듈 재사용)
- 문서 메인 목록보다 **송금 상세/내역에서 즉시 열 수 있는 경험**을 우선한다.

## 7F. 안전정책(SafePay)

- 허용 목록 기반 송금
- 수신자 변경 후 24시간 송금 제한(쿨다운)
- 고액 추가 확인(예: 평소의 2배 또는 특정 금액 이상)
- 중복 전송 방지(중복 방지 키)
- 차단 시에는 "왜 막혔는지 / 언제 다시 가능한지"를 사용자 언어로 설명해야 한다.

## 7G. 보관/이자(Vault, 데모 시뮬레이션)

**"남는 돈을 따로 보관하고, 필요하면 다시 꺼내 쓸 수 있게"**

> 보관/이자는 DonDone의 메인 메시지를 해치지 않는 선에서,
> `남는 돈 보관 -> 예상 이자 확인 -> 송금 연결` 경험을 제공하는 데모 시뮬레이션 기능이다.

### 자금 순환 구조(데모 설명용)
```text
사용자 잔액
   ->
보관 금액 설정
   ->
예상 이자 계산(시뮬레이션)
   ->
필요 시 다시 송금/생활비로 전환
```

### V1. 보관 잔액 표시(P0, 시뮬레이션)
- 현재 보관 중인 금액
- 보관 가능 금액
- 송금 가능 전환 금액

### V2. 예상 이자 표시(P0, 시뮬레이션)
- 일별/월별 예상 이자
- 연 환산 기준값 표시
- 단, **수익 보장처럼 보이는 표현 금지**

### V3. 목표 저축(P1)
- 목표(학비/집 수리/귀국 자금) 설정 + 급여에서 자동 적립
- 진행률 바 + 예상 달성일
- 달성 시 "바로 송금" 옵션 연결

### V4. 실제 연동(P1)
- 메인넷 연동, 실 yield 파트너십 검토

---

## 7H. 문서(Documents): 증빙 리포트 / 근거 자료 묶음 / 정산 문제 해결

### D1. 증빙 리포트(P0)
- 월 단위(또는 사건 단위)로 생성
- 필수 포함:
  - 월간 요약 + WorkProof 상세 표
  - 급여 추정 근거 + 차액 리포트
  - 수정 이력(감사 로그) 표
  - 첨부 목록(있으면)

### D2. 근거 자료 묶음(P0)
- 증빙 리포트 + 제출용 요약 + 체크리스트
- 출력 형태: PDF(기본) / ZIP(선택, 첨부가 많은 경우)

### D3. 신고 준비(Instant Claim v0) (P0)
- 정산 문제 해결 흐름의 마지막 단계로 제공한다.
- 자동 제출이 아니라 반자동 지원:
  - 접수 경로 안내(온라인/전화/방문)
  - 제출용 자동 요약 문장 생성(복사 버튼)
  - 근거 자료 묶음 공유(메일/메신저/저장)

> 원칙: 첫 단계는 곧바로 신고가 아니라 **확인과 설명 요청**이고, 그래도 해결되지 않을 때 증빙과 외부 경로로 이어진다.

## 7I. 고용주 기능(Employer Support, P0 지원 기능)

### 핵심 원칙
- 고용주 기능은 **평가 도구**보다 **운영 리스크를 줄이는 도구**로 설계한다.
- `성실도 점수`, `자동 선별 점수`처럼 사람을 줄 세우는 표현은 쓰지 않는다.
- Advance를 안전하게 열고, 정산/분쟁 가능성을 미리 보게 하는 보조 기능으로 설명한다.

### E1. 운영 콘솔(P0)
- 오늘 출근/퇴근 현황
- 누락 기록 / 수정 요청 / 위치 불일치
- 월말 정산 미리보기
- 급여 차액 가능성 알림
- 근로자 확인 요청 상태와 1차 확인/정정 필요 건 분류
- 선지급 가능 대상 확인
- 선지급 운영 시 회사가 감당해야 할 정산 리스크 가시화

### E2. Verified Worker Summary(P0)
- P0에서는 공유용 패스포트가 아니라 **운영 콘솔 내부에서만 보는 설명 가능한 요약 카드**로 제공한다.
- 포함 가능한 항목 예시:
  - 재직 확인
  - 최근 근무기록 일관성 높음
  - 최근 정산 이슈 없음
  - 선지급 후 정상 정산 이력
  - 최근 계약 상태 정상

### E2-1. Work Passport(P1 확장)
- `Work Passport`라는 이름은 **공유 가능성과 검증 가능성**이 붙을 때 사용한다.
- 권장 형태:
  - 근로자 동의 기반 PDF
  - 링크 또는 QR 기반 검증
  - 재고용/이동 시 참고자료 제출
- 즉, P0 내부 요약 카드를 그대로 `패스포트`라고 부르지 않는다.

### E3. 사용 가능한 신호와 사용하지 않는 신호(P0 원칙)
- 사용할 수 있는 신호:
  - 출퇴근 일관성
  - 잦은 수정/누락 여부
  - 정산 이슈 발생 여부
  - 선지급 후 정상 정산 완료 여부
- 사용하지 않는 방식:
  - 채용 자동판단 점수
  - 성실도 랭킹
  - 사람을 직접 줄 세우는 점수 노출

### E4. 왜 P0에 포함하는가
- Advance는 근로자 경험만으로 설명하기보다, **고용주 입장에서 운영 리스크가 낮아진다**는 이야기가 같이 있어야 설득력이 커진다.
- 또한 회사 입장에서 **현금흐름 부담을 크게 늘리지 않으면서 복지로서 선지급을 제공할 수 있다**는 설명이 있어야 도입 이유가 더 분명해진다.
- 따라서 고용주 기능은 별도 메인 제품이 아니라, **미리받기와 정산 기능을 안전하게 보여 주기 위한 P0 지원 기능**으로 둔다.
- 이때 P0의 핵심 산출물은 `Verified Worker Summary`이고, 외부 공유형 `Work Passport`는 P1로 넘긴다.

## 8. 인공지능(코파일럿) 설계

### 8.1 목적: “자유대화”가 아니라 “현재 화면을 돕는 설명/문장 생성”
- 코파일럿은 사용자가 지금 보고 있는 화면을 **이해하고 다음 행동을 정리하도록 돕는 보조 기능**이다.
- 각 핵심 화면에 **"이 화면 설명해줘"** 버튼 제공
- 추천 질문 칩을 함께 제공해 사용자가 무엇을 물어볼지 바로 선택할 수 있게 한다.

### 8.2 코파일럿 UX(버튼/칩 예시)
#### (1) 금융 홈
- 버튼: `이 화면 설명해줘`
- 추천 질문 칩:
  - `지금 내가 할 수 있는 행동은 뭐야?`
  - `왜 미리받기 금액이 이렇게 나왔어?`
  - `보관과 송금 중 뭐가 맞아?`

#### (2) 차액 감지 화면
- 버튼: `이 화면 설명해줘`
- 추천 질문 칩:
  - `왜 차액이 생겼어?`
  - `어떤 근거로 그렇게 판단했어?`
  - `지금 내가 할 수 있는 다음 행동은 뭐야?`

#### (3) 송금 차단/보호 화면(SafePay)
- 버튼: `왜 송금이 막혔어?`
- 칩:
  - `24시간 쿨다운이 뭐야?`
  - `얼마부터 추가 확인이야?`
  - `다시 보내려면 어떻게 해?`

#### (4) 신고 준비 화면(Instant Claim)
- 버튼: `제출용 문장 만들어줘`
- 칩:
  - `짧게 요약해줘`
  - `정중한 문장으로 써줘`
  - `영어로 바꿔줘`

### 8.3 코파일럿이 제공하는 기능(권장 3종, P0)
> 코파일럿은 **설명/번역/문장 생성만** 수행한다.  
> 숫자 계산/판정은 서버 룰 기반 결과를 그대로 사용한다.

1) **돈 상태, 차액, 정책 차단 이유를 쉬운 말로 설명**한다.
2) **제출용/공유용 요약 문장**을 만든다.
3) **화면 facts 기반 번역과 쉬운 표현 변환**을 제공한다.

### 8.4 가드레일(환각/오답 방지 정책) — PRD 고정
- 코파일럿은 **새로운 숫자를 계산하거나 추정하지 않는다.**
- 코파일럿은 서버가 제공한 **facts(사실 데이터)**만 사용해 문장을 만든다.
- facts 예시:
  - 차액 금액, 근무시간 합계, 연장/야간 시간, 해당 WorkProof ID 목록, 수정 사유 목록, SafePay 차단 코드, 허용 목록 상태, 미리받기 가능 금액 근거 등
- facts 밖의 내용을 말하려고 하면:
  - "확인할 수 없음/추정 불가"로 응답하고, 필요한 입력을 안내한다.
- 모든 코파일럿 응답에는 화면 하단에 공통 문구를 표시:
  - "이 안내는 화면에 표시된 사실을 바탕으로 작성된 참고 문장입니다."

### 8.5 RAG 정책: “필수 아님, 필요 시 RAG-lite”
- 대부분 질문은 화면 facts로 해결한다.
- 용어/기능 설명(예: 증빙 리포트 vs 근거 자료 묶음, 보관/이자 범위)이 더 필요할 때만
  짧은 FAQ/용어집을 **키워드 검색 또는 소규모 임베딩 Top-k**로 제공한다.
- 대규모 지식 기반 챗봇은 P1 이후로 미룬다.

## 9. 데모 전용 기능: 타임 트래블(Time Travel)

> **실사용 기능이 아니라 발표 데모를 위한 장치**이며, 릴리스 빌드에서는 비활성화한다.

### 9.1 목적
- 실제 한 달 사용 흐름을
  **근무 -> 반영 -> 미리받기 -> 급여 점검 -> 차액 감지 -> 송금/보관** 순서로
  **15~30초 안에 압축해서 보여주기**

### 9.2 동작 방식(권장)
- 데모 시작 시 **시드(고정) 데이터**를 주입한다.
  - WorkProof 기록(일자별 출퇴근 + 일부 수정 이력 포함)
  - 미리받기 가능 금액 변화
  - 급여 이벤트(예상/실제 지급 결과 반영)
  - 송금 이벤트(성공/실패 1건씩)
  - 보관/이자 시뮬레이션 이벤트
- 사용자는 화면 상단의 슬라이더/재생 버튼으로 `기준일(asOf: YYYY-MM-DD)`을 바꾼다.
- 앱은 기준일 이전 데이터만 반영해 다음 화면을 즉시 갱신한다.
  - 홈 상태, 월간 요약, 참고용 추정, 차액 감지, 추천 행동, 미리받기 가능 금액, 송금/보관 상태

### 9.3 구현 원칙(데모 품질을 위한 기준)
- **"시드 데이터는 고정"**이지만, **집계/계산/판정 로직은 실제 로직**을 사용한다.
- 기준일 변경은 "데모 모드 파라미터"로 서버에 전달한다.
  - 예: 요청 헤더 `X-Demo-AsOf: 2026-03-20`
- 슬라이더만 움직여도 `쌓인다 -> 쓴다 -> 확인한다 -> 문제 시 보호된다` 흐름이 자연스럽게 이어져야 한다.

### 9.4 데모 모드 정책
- 데모 전용 계정에서만 활성화
- 일반 계정/릴리스 빌드에서는 완전 비활성(설정 숨김)

## 10. 서버 아키텍처(Feature-first) — PRD 구현 가이드

> 이 장은 제품 소개용이 아니라 **구현 기준을 맞추기 위한 팀 내부 가이드**다.

### 10.1 구성 요소(한 대의 서버 + Compose 컨테이너)
- `api-server`(Spring): 외부 공개 API, 모든 비즈니스 규칙
- `pdf-service`(Node): HTML 템플릿 + 브라우저 렌더링으로 PDF 생성(내부 전용)
- `postgres`: 저장소(상태/로그/잡)
- `redis`: 쿨다운/락/보조 큐(권장)
- `s3`: 문서 저장(Private) + presigned URL 다운로드

### 10.2 코드 구조(Feature-first 권장)
```text
com.dondone
  ├─ auth
  ├─ employer
  ├─ home
  ├─ workproof
  ├─ advance
  ├─ wage
  ├─ safepay
  ├─ remittance
  ├─ vault
  ├─ documents
  ├─ jobs
  ├─ notifications
  └─ shared
```

각 기능 모듈 내부는 MVP에서는 단순하게:
- `api`(컨트롤러/요청응답 DTO)
- `service`(유스케이스/정책)
- `repo`(JPA)
- `adapter`(외부 연동)
- `model`(엔티티/도메인 모델)

추가 책임:
- `employer`: 회사 확인 코드, 고용주-근로자 연결, 운영 콘솔, Verified Worker Summary를 담당한다.
- `jobs`: PDF, 송금, Summary 생성, 무결성 앵커 기록 같은 비동기 작업을 담당한다.

### 10.3 외부 의존성은 인터페이스로 숨긴다(팀 협업 핵심)
블록체인, 회사 확인, 파일 저장, PDF, 인공지능은 아래 인터페이스 뒤에 둔다.

#### (1) 블록체인 게이트웨이(remittance)
- 팀원은 `TransferUseCase`만 사용
- web3j 사용/nonce/gas는 adapter에서만

```java
public interface TransferUseCase {
  TransferId requestTransfer(RequestTransferCommand cmd); // Idempotency-Key 포함
  TransferDto getTransfer(TransferId id);
}

public interface Erc20Gateway {
  TxHash submitTransfer(String tokenAddress, String toAddress, BigInteger amountAtomic);
  Optional<TxReceipt> getReceipt(TxHash txHash);
}
```

#### (1-a) 무결성 앵커(integrity)
- 블록체인은 **평가 엔진**이 아니라 **무결성 증빙 인프라**로 사용한다.
- 권장 활용:
  - 근로계약서 해시 저장
  - 월별 근무기록 해시 저장
  - 회사 확인 시점 타임스탬프
  - Proof Pack 해시 저장
- 지양:
  - 원본 출퇴근 로그 온체인 저장
  - 개인정보 직접 저장
  - 체인 기반 성실도 점수/랭킹

#### (2) PDF 렌더러(documents)
```java
public interface PdfRenderer {
  byte[] renderProofPack(ProofPackPayload payload);
  byte[] renderClaimKit(ClaimKitPayload payload);
  byte[] renderTransferReceipt(ReceiptPayload payload);
}
```

#### (3) 파일 저장(S3)
```java
public interface FileStorage {
  StoredFile put(String key, byte[] bytes, String contentType);
  URI presignGet(String key, Duration ttl);
}
```

#### (4) 코파일럿(인공지능)
```java
public interface CopilotService {
  CopilotAnswer explainWithFacts(CopilotRequest request); // facts를 입력으로 받음
  CopilotAnswer draftClaimSummary(CopilotRequest request);
  CopilotAnswer translateWithFacts(CopilotRequest request);
}
```

#### (5) 고용주 확인(employer)
```java
public interface EmployerVerificationGateway {
  Optional<EmployerVerificationResult> verifyCode(String verificationCode);
}
```

### 10.4 비동기 잡(jobs) 설계(송금/문서 생성 안정화)
- PDF 생성, 송금 전송/확정 확인은 "요청-응답"에서 분리한다.
- `jobs` 테이블로 상태/재시도를 관리한다.

#### 잡 상태
- QUEUED -> RUNNING -> DONE / FAILED

#### 잡 종류(예시)
- RENDER_PROOF_PACK
- RENDER_CLAIM_KIT
- SUBMIT_TRANSFER
- POLL_TRANSFER_RECEIPT
- RENDER_TRANSFER_RECEIPT
- BUILD_WORKER_SUMMARY
- ANCHOR_INTEGRITY_HASH

### 10.5 송금이 비동기 처리인 이유(요약)
- 전송은 빠를 수 있지만, **완료 확정은 전송 결과(영수증) 확인이 필요**
- 네트워크 재시도/중복 요청이 흔해 **중복 방지 키와 재시도 정책**이 필수
- nonce 충돌을 피하기 위해 **전송 잡은 직렬화(동시성=1)**가 MVP에 안전

### 10.6 데이터 모델(개요)
- users, workplaces, contracts
- employers, employer_links, employer_verifications
- workproof_records + workproof_modifications(수정 이력) + workproof_attachments
- advance_snapshots, advance_requests
- wage_estimates, wage_verifications
- recipients, transfers
- vault_deposits, vault_yield_logs
- documents(ProofPack/ClaimKit/Receipt) + document_links(관련 엔티티 연결)
- worker_summaries, integrity_anchors
- jobs
- events(감사 로그, append-only)

> 핵심: "수정 이력/첨부/근거"는 문서에 자동 포함되므로, DB에서도 분리 저장해 추적 가능해야 한다.
> `work_passports`는 P1 공유·검증 기능이 확정될 때 별도 스키마/API로 추가한다.

## 13. 3주 발표 가능 최소본 실행 계획(권장, Jira 기반)

> 이 계획은 **3주 안에 발표 가능한 최소본**을 만드는 기준이다.  
> 범위표의 `P0(6주 Demo MVP)`는 더 넓은 목표이고, 아래 계획은 그중 발표에 필요한 최소 흐름을 먼저 완성한 뒤 6주 MVP로 확장하는 순서다.

### 13.0 인원 구성
- 풀스택: 강보승
- 풀스택: 차지훈
- 서버/인프라: 장현준
- 서버: 정관우
- 블록체인: 최성민
- 블록체인: 이건도

### 13.1 P0 포함 범위
- **WorkProof**: 출근/퇴근 기록, 근무 상세, 근무 수정(W4: 수정 사유 필수 + 선택 증거)
- **Advance**: 검증 완료 근무 + 급여 흐름/차단 규칙 기반 한도 계산, 차단 사유, 신청/이력 시뮬레이션
- **Employer Support**: 운영 콘솔, 회사 확인 코드, 선지급 가능 대상 확인, Verified Worker Summary
- **WageShield**: 월간 요약/예상(룰 고정), 입금 입력, 차액 감지(근거 포함)
- **Evidence**: Proof Pack/Claim Kit PDF 생성(수정 이력 포함)
- **Remit**: 테스트넷 송금, tx 상태 추적, 영수증 + SafePay 최소
- **Instant Claim v0**: 반자동 제출 준비(체크리스트/문장/경로 안내)
- **Demo Time Travel**: seed + asOf로 상태 렌더(데모용)
- **Chat Copilot(P0)**: "이 화면 설명해줘" + 추천 질문 칩(화면 facts 기반)

> 주의: 추가 메시지성 기능은 현재 제품/UX 기준에서는 **핵심 데모 동선보다 우선순위가 낮다.**
> Jira에는 `stretch` 또는 `optional` 라벨로 별도 관리할 수 있다.

### 13.2 Jira 운영 원칙

#### 이슈 계층(권장)
- **Epic**: 기능 축 단위
- **Story**: 실제 구현/추적 단위

> 권장 원칙: **Epic은 기능 기준**, **Story는 실제 작업 기준**으로 관리한다.  
> 즉, Epic에 `[ANDROID]`, `[BACKEND]`, `[BLC]`를 직접 붙이기보다  
> `[WRK]`, `[WAGE]`, `[DOC]`, `[BLC]`, `[INFRA]`처럼 **기능/영역 기준**으로 만들고,  
> Story는 prefix 없이 **바로 실행할 수 있는 작업 제목**으로 쓰는 편이 관리하기 쉽다.

#### Prefix 규칙(권장)
- **Epic prefix**: 기능/도메인 기준, 대문자 고정
  - 예: `[AUTH]`, `[WRK]`, `[ADV]`, `[WAGE]`, `[DOC]`, `[BLC]`, `[INFRA]`, `[COP]`
- **Story 제목**: prefix 없이 실제 구현 작업 기준으로 작성
  - 예: `Jenkins 빌드 파이프라인 구성`
  - 예: `실제 지급 결과 확인 화면 구현`
  - 예: `근거 자료 묶음 PDF 생성 기능 구현`

#### 워크플로우(권장)
- `Backlog -> Selected for Dev -> In Progress -> In Review -> Done`
- 장애/대기 상황은 `Blocked` 라벨로 관리

#### 라벨(권장)
- `p0`, `p1`, `frontend`, `backend`, `infra`, `blockchain`, `demo`, `copilot`, `pdf`, `time-travel`, `safepay`, `stretch`

#### Definition of Done(권장)
- 프론트: loading / empty / error / success 상태 중 필요한 상태가 보인다.
- 서버: API DTO/validation/logging이 정리되어 있다.
- 블록체인: tx hash / 상태 / 실패 사유가 재현 가능하다.
- 데모: seed/reset/asOf로 재현 가능하다.
- 문서: 생성/다운로드/링크 확인이 가능하다.

### 13.4 1주차: 기본 플로우 뼈대 + 테스트넷 송금 PoC + 데모 모드 틀

#### 주말까지 완료 기준
- 출근/퇴근 기록 생성·조회가 앱에서 된다.
- 테스트넷 송금 1회 성공(tx hash 확인).
- asOf 날짜를 바꾸면 화면이 일부라도 바뀐다(타임트래블 틀).

#### 사람별 작업
- **강보승(풀스택)**
  - 홈/월간요약 UI 틀(이번 달 근무시간/예상급여 자리)
  - 입금 입력 화면/차액 화면 UI 틀(동선 고정)
  - 타임트래블(날짜 슬라이더/재생) UI 틀
- **차지훈(풀스택)**
  - 공통 UI 컴포넌트(로딩/에러/토스트/상태 배지)
  - 근무 목록/상세 UI(시간/위치/메모 기본)
- **장현준(서버/인프라)**
  - CI/CD, dev/demo 환경 분리, env/secret, requestId 로그
  - PDF 생성 방식 PoC(템플릿 1장 출력 성공)
- **정관우(서버)**
  - WorkProof API/DB(근무 생성/조회)
  - 월간 집계/계산 엔진 스켈레톤(총시간/야간/연장 구분 정도)
  - 회사 확인 코드/재직 확인 데이터 구조 초안
- **최성민(블록체인)**
  - 지갑 연결 + 테스트넷 전송 PoC(tx 생성/서명/전송)
  - tx 상태 확인 방식 결정(우선 폴링)
- **이건도(블록체인)**
  - 무결성 해시 규격 초안(해시 대상 필드/메타 구조)
  - 데모 시드 데이터 포맷(JSON) 초안(정상 케이스 1)

### 13.5 2주차: 차액 감지 -> Proof Pack -> 송금/영수증 연결 + 근무 수정(W4)

#### 주말까지 완료 기준
- 타임트래블로 급여일까지 이동 -> 입금 입력 -> 차액 감지 -> Proof Pack 생성 -> 송금 -> 영수증까지 한 번에 된다.

#### 사람별 작업
- **강보승(풀스택)**
  - 입금 입력 UX + 차액 결과 화면(원인 카드/근거 링크)
  - 차액 이후 액션 UI(Proof Pack / Claim 준비 / 신고 준비)
  - 고용주 운영 콘솔 화면 틀(출퇴근 현황/누락/차액 가능성)
  - SafePay 관련 UI(차단/추가 확인/안내 화면)
- **차지훈(풀스택)**
  - Proof Pack 생성 UX(진행/완료, "빠르게 생성" 연출)
  - 영수증 화면(Explorer 링크/tx hash/상태) + 공유 버튼
- **장현준(서버/인프라)**
  - Proof Pack PDF 생성 MVP(근무 요약 + 생성 메타)
  - 파일 저장/다운로드 경로 구성(간단 스토리지 또는 URL)
- **정관우(서버)**
  - 근무 수정(W4): 수정 사유 필수 + 선택 증거 메타 저장
  - 차액 감지 로직 + 사유(룰 기반) + 근거 DTO(WorkProof 리스트/표)
  - Verified Worker Summary DTO 초안
  - `/demo/seed`, `/demo/state?asOf=` 구현(시드 주입 + 날짜 기준 집계)
- **최성민(블록체인)**
  - 송금 E2E(전송 -> 상태 -> 영수증 데이터)
  - SafePay 최소 적용(allowlist/쿨다운/고액확인/idempotency)
- **이건도(블록체인)**
  - Proof Pack 해시 생성(오프체인) + QR payload 확정
  - 데모 시드 "차액 케이스" 1개 추가(의도된 누락/차액)

### 13.6 3주차: Instant Claim v0 + 챗봇(P0) + Proof Pack QR 검증 + 안정화

#### 최종 완료 기준
- PRD 6-step 데모 완주
- Proof Pack QR 검증 화면에서 `OK`
- 코파일럿이 "왜 차액?"를 facts-only로 설명한다.
- Instant Claim은 반자동(사용자 제출) 문구가 고정되어 있다.

#### 사람별 작업
- **강보승(풀스택)**
  - Instant Claim v0 화면 플로우(유형 선택 -> 키트 생성 -> 경로 안내)
  - 코파일럿 UI("이 화면 설명해줘" + 추천 칩 5개)
  - 데모 동선(타임트래블 -> 차액 -> 팩 -> 송금 -> 클레임 -> QR/챗봇 하이라이트) 고정
- **차지훈(풀스택)**
  - 제출 체크리스트/안내/문구 폴리싱(반자동 강조)
  - 영수증/문서/검증 화면 폴리싱(시각 임팩트)
  - Verified Worker Summary 카드 UI 폴리싱
- **장현준(서버/인프라)**
  - E2E 테스트 2개(정상/차액) + 버그 배시 운영
  - 데모 모드 재현 스크립트(seed/reset 한 번에)
- **정관우(서버)**
  - Claim Kit 생성(템플릿 v1: 일자별/수정사유 포함 + 무결성 섹션 포함)
  - 코파일럿 API 2개 핵심:
    - `/ai/explain` (차액/정책 사유 설명)
    - `/ai/claim-summary` (제출용 요약 문장)
  - facts-only 가드레일: "숫자 새로 계산 금지 / 서버 facts만 사용"
  - 추가 안내 메시지 기능은 필요 시 `stretch`로 별도 관리
- **최성민(블록체인)**
  - 실패 케이스 2개 고정(거절/네트워크 등) + 상태 표준화
  - SafePay 차단 사유 코드/로그 정리(프론트 표시와 연결)
  - 근로계약서/Proof Pack 해시 앵커링 데모 흐름 정리
- **이건도(블록체인)**
  - QR Verifier 웹(검증 OK/FAIL, 생성일/포함 건수/해시 표시)
  - 발표용 검증 성공 장면 스크립트/데이터 고정

### 13.8 Jira 운영 팁
- Epic은 기능 축으로 끊고, Story는 prefix 없이 실제 구현 단위로 쓴다.
- 기술 구분은 Story 제목이 아니라 담당자, 컴포넌트, 라벨로 관리하는 것을 권장한다.
- Story 하나는 가능하면 **한 사람이 책임지고 완료 여부를 판단할 수 있는 수준**으로 자른다.
- `Blocked` 사유는 반드시 외부 의존성(API, RPC, PDF, env) 중 무엇인지 명시한다.
- 매주 말에는 **"데모를 실제로 끝까지 돌릴 수 있는가"**를 기준으로 Done 여부를 판단한다.

## 14. 리스크와 대응

> 리스크 관리는 사용자를 강하게 묶는 방향이 아니라, **위험한 상황은 처음부터 덜 열어주는 방향**을 기본 원칙으로 한다.

| 리스크 | 영향 | 대응 |
|---|---|---|
| "근무 앱처럼 보인다"는 첫인상 | 제품 정체성 혼선 | 홈/금융 우선 구조, "이번 달 내 돈" 중심 카피 적용 |
| "미리받기/보관"을 실서비스로 오해 | 신뢰/규제 리스크 | 데모 시뮬레이션/테스트넷 문구 고정, 과장 금지 |
| "자체 신용평가 서비스"처럼 보임 | 데이터/설명책임 과대 기대 | 검증된 근무 + 회수 가능성 확인 구조로 표현, 범용 점수 서사 제거 |
| 고용주 기능이 채용 점수처럼 보임 | 설명책임/법적 부담 증가 | 운영 콘솔 + 내부 요약 카드로 제한, 자동 판단 점수 미도입 |
| "수정 가능 = 조작" 피드백 | 신뢰 하락 | 수정 사유 필수 + 감사 로그 + 문서 자동 반영으로 "설명 가능한 기록" 구현 |
| 마지막 급여 주기 직전 선지급 | 회수 실패 위험 | 계약 종료 30일 이내 경고/축소, 14일 이내 또는 마지막 급여 주기 진입 시 신규 차단 |
| 급여 계산 정확도 논란 | 불필요 경보/불신 | "참고용 추정" 명시 + 이상 탐지 중심 + 근거 제시 + 근로자 직접 신고와 회사 확인 역할 분리 |
| 코파일럿 오답/환각 | 잘못된 안내 위험 | facts 기반만 허용, 숫자 계산 금지, 추정 금지 |
| 체인/RPC 불안정 | 송금 데모 실패 | 상태머신/재시도/직렬화, 데모 토큰 사용 |
