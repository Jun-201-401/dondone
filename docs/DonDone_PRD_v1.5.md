# PRD v1.5 — DonDone
> **"당신의 돈(Don), 처음부터 끝(Done)까지 던던하게."**
> 근무 기록을 단순 기록으로 끝내지 않고, **금융 행동의 근거**로 연결해
> **미리받기 · 월급 확인 · 테스트넷 송금 · 보관/이자 시뮬레이션**까지 한 흐름으로 제공하는 앱

## 0. 처음 보는 사람을 위한 설명

### 0.0 브랜드 표기 원칙(v1.5)
- 서비스명 표기는 **DonDone**을 기본으로 사용한다.
- 발표/데모 화면의 핵심 슬로건은 아래 문구를 고정 사용한다.
  - **"당신의 돈(Don), 처음부터 끝(Done)까지 던던하게."**
- 기술 맥락에서 기존 프로젝트명(`WorkProofPay`)이 남아 있어도, 대외 노출 텍스트는 DonDone 기준으로 통일한다.

### 0.1 한 문장 요약
> **"일한 기록이 내 돈의 근거가 되어, 급하면 먼저 받고, 이번 달 월급을 확인하고, 가족에게 보내고, 남는 돈은 따로 보관할 수 있는 앱."**

### 0.2 이 제품이 해결하려는 핵심 문제
외국인 노동자는 다음 문제가 동시에 발생한다.
- **급여일 전 현금흐름 공백**: 이미 일했지만 급여일 전까지 쓸 수 있는 돈이 부족할 수 있다.
- **급여 확인 어려움**: 연장·야간·공제 등으로 "내가 받아야 할 돈"과 "실제로 받은 돈"이 다를 수 있는데, 무엇이 빠졌는지 알기 어렵다.
- **송금 불편**: 소액을 자주 보내면 수수료 부담이 크고, 속도와 영수증 관리도 번거롭다.
- **남는 돈 관리의 부재**: 남는 돈을 따로 보관하거나 다음 송금 전까지 관리할 수 있는 선택지가 흩어져 있다.
- **근무 근거 부족**: 문제가 생겼을 때 "내가 일했다"는 기록이 약하면 보호 기능과 금융 기능 모두 신뢰가 떨어진다.

### 0.3 우리가 만드는 해법의 핵심
- DonDone은 **근무 기록 앱처럼 보이는 것보다, 근무를 바탕으로 금융 행동이 열리는 앱처럼 보여야 한다.**
- 근무 기록은 별도 목적이 아니라 **미리받기/월급 확인/문서의 신뢰 근거**가 된다.
- 홈에서는 기록보다 먼저 **"이번 달 내 돈 상태"와 "다음 행동"**을 보여준다.
- 미리받기와 송금은 사용자가 바로 체감하는 전면 가치이고, 월급 확인과 문서는 보호 기능으로 뒤를 받쳐준다.
- 보관/이자는 테스트넷/데모 범위에서 **생활형 자산 관리 경험**으로 소개하되, 투자/수익 보장처럼 보이지 않게 설계한다.

## 1. 문제 정의

### 1.1 돈의 흐름에서 발생하는 문제(전체 구간)
```text
일한다 ──→ 근무가 반영된다 ──→ 급여 전 현금 필요 ──→ 급여 받는다 ──→ 생활비/가족송금 ──→ 남는 돈 보관
  │               │                      │                    │                     │
  ▼               ▼                      ▼                    ▼                     ▼
기록 약함         반영 상태 불투명         미리 쓸 돈 부족        누락/체불 확인 어려움   보관/이자/영수증 관리 분산
```

### 1.2 기존 대안의 한계
| 기존 방식 | 되는 것 | 안 되는 것 |
|---|---|---|
| 근태관리 앱 | 출퇴근 기록 | 돈의 흐름(미리받기/월급 확인/송금)과 직접 연결되지 않음 |
| 급여 계산기 | 월급 계산 | 근무 근거와 연결이 약하고, 실제 행동(문서/공유/다음 금융 액션)으로 이어지지 않음 |
| 송금 앱/은행 | 송금 | 내가 일한 기록과 월급 확인, 이후의 근거 관리와 연결되지 않음 |
| 개인 지갑 앱 | 코인 전송/보관 | 외국인 사용자 관점의 안전정책·생활형 UI·근무 기반 신뢰가 부족함 |
| 가불/대출 서비스 | 현금 확보 | 근무 반영 데이터와 급여 보호 기능이 한곳에 있지 않음 |

## 2. 타깃 사용자

### 2.1 MVP 타깃
- 제조/건설/농축산/서비스 등 현장 중심 업종의 외국인 근로자
- 시급/일급/월급 혼재, 연장/야간 가능성 존재
- 한국어 급여 문서 이해가 어렵고, 월급 확인이 번거로운 사용자
- 매월 소액 송금이 잦고, 급여일 전 현금흐름 압박을 자주 느끼는 사용자

### 2.2 핵심 사용자 목표(JTBD)
1) **일한 만큼 지금 할 수 있는 금융 행동이 늘어나는 걸 보고 싶다.**
2) **급하면, 반영된 근무 기록을 바탕으로 일부를 먼저 받고 싶다.**
3) **월급이 이상한지(누락/지연/공제 과다)를 쉽게 알고 싶다.**
4) **가족에게 빠르고 저렴하게 보내고 싶다.**
5) **남는 돈을 따로 보관하고, 가능하면 조금이라도 불리는 경험을 해보고 싶다.**

## 3. 제품 목표

### 3.1 Demo MVP에서 반드시 보여줄 것(P0)
- **금융 홈(Money Home)**: 이번 달 내 돈 상태 + 다음 행동 추천 + 빠른 금융 CTA
- **근무 기록(근거화)**: 출근/퇴근 기록 + 수정 사유 필수 + 수정 이력 자동 저장
- **미리받기(데모 시뮬레이션)**: 반영된 근무 기반 가능 금액 + 신청 + 상환 예정일
- **급여 점검(이상 탐지)**: 참고용 급여 추정 + 실제 입금 비교 + 차액 근거 제시
- **차액 이후 액션**: 증빙 리포트 / 근거 자료 묶음 / 신고 준비 / 제출용 요약 문장
- **테스트넷 송금**: 수신자 허용 목록 + 전송 + 상태 추적 + 영수증
- **보관/이자(데모 시뮬레이션)**: 스테이블코인 보관 잔액 + 예상 이자 + 송금 연결
- **문서(PDF)**: 증빙 리포트, 근거 자료 묶음, 송금 영수증을 저장하고 앱에서 공유/다운로드

### 3.2 성공 기준(데모 관점)
- 앱을 켰을 때 5초 안에 "근무 기록 앱"보다 **"돈 관리 앱"**이라는 인상이 들어야 한다.
- 30초 안에 **일한 만큼 쌓인다 -> 먼저 받는다/확인한다/보낸다/보관한다** 흐름이 이해돼야 한다.
- 3분 데모에서 "한 달 여정"을 **15~30초로 압축해서 보여주는 데모 장치(Time Travel)**로 스토리를 완주한다.
- "차액 감지 -> 근거 확인 -> 문서 생성 -> 신고 준비"까지 **끊기지 않는 보조 흐름**을 구현한다.

## 4. 범위(P0/P1) 확정표

> 원칙: P0는 **금융 중심으로 확실히 되는 데모**, P1은 **정교화/실서비스 확장**이다.  
> 미리받기/보관은 P0에서 **데모 시뮬레이션**으로 보여주고, 실제 정책/실거래 연결은 P1로 미룬다.

| 영역 | P0(6주 Demo MVP) | P1(후속) |
|---|---|---|
| 금융 홈 | 이번 달 내 돈 상태, 다음 행동 추천, 빠른 이동 | 개인화 추천, 리마인드/알림 고도화 |
| 근무 기록 | 출근/퇴근 원탭, 수정 사유 필수, 첨부 선택, 수정 이력 자동 포함 | 지오펜스 기반 자동 알림, 다중 근무지 |
| 미리받기 | **데모 시뮬레이션**: 가능 금액, 신청, 상환 예정일, 근거 설명 | 실제 정책/파트너 연동, 한도 고도화 |
| 급여 점검 | **참고용 추정 + 이상 탐지**(근거 포함), 공제 미입력 배지/임계값 완화 | 급여명세서/계약서 자동 추출 고도화 |
| 송금 | 테스트넷 전송, 상태 추적, 영수증, SafePay | 정기 송금/자동 송금, 한도 고도화 |
| 보관/이자 | **데모 시뮬레이션**: 잔액, 예상 이자, 송금 전환 | 메인넷 연동, 실 yield 파트너십 |
| 문서(PDF) | 증빙 리포트/근거 자료 묶음/송금 영수증 PDF(템플릿 v1) | 문서 템플릿 확장, 다국어 템플릿 |
| 신고 준비 | Instant Claim v0(반자동): 안내/체크리스트/자동 문장/공유/바로가기 | 기관 제출 자동화 또는 제휴 연동 |
| 인공지능(코파일럿) | 화면 설명/번역/요약 문장 생성 3종 + facts 기반 가드레일 | RAG-lite 고도화, 언어 확장 |
| RAG | **필수 아님**. 기본은 화면 facts | FAQ/용어집 기반 RAG-lite |

## 5. 핵심 용어(이 문서에서 사용하는 말)

| 용어 | 설명 |
|---|---|
| 금융 홈(Money Home) | 이번 달 내 돈 상태, 다음 행동, 금융 진입점을 보여주는 메인 홈 |
| 근무 증거(WorkProof) | 출퇴근 시점의 시간/위치 스냅샷 + 사용자 확인으로 만든 "근무 기록 1건" |
| 근무 반영 | 기록된 WorkProof가 미리받기/월급 확인에 사용 가능한 상태로 집계되는 것 |
| 미리받기(Advance) | 반영된 근무 기준으로 급여일 전 일부를 먼저 받는 기능(데모에서는 시뮬레이션) |
| 급여 점검(Wage Shield) | 참고용 급여 추정과 실제 입금을 비교해 "이상 징후"를 잡고 근거를 제시하는 기능 |
| 증빙 리포트 | 근무 로그 + 계산 근거 + 차액 리포트를 한 파일(PDF)로 묶은 문서(내부명: Proof Pack / `PROOF_PACK`) |
| 근거 자료 묶음 | 제출/상담에 필요한 요약·근거·첨부를 묶은 문서(PDF 또는 ZIP)(내부명: Claim Kit / `CLAIM_KIT`) |
| 신고 준비(Instant Claim v0) | 자동 제출이 아니라, 제출을 쉽게 만드는 반자동 지원 기능 |
| 보관/이자(Vault) | 남는 스테이블코인을 따로 보관하고 예상 이자를 확인하는 기능(데모 시뮬레이션) |
| 허용 목록 | 송금은 등록된 수신자에게만 가능하도록 제한하는 목록 |
| 중복 방지 키(Idempotency Key) | 같은 요청이 여러 번 와도 1번만 처리하도록 하는 요청 키(송금/문서 생성에 필수) |
| 전송 해시(Tx Hash) | 블록체인 전송의 고유 ID(상태 추적/영수증에 포함) |

## 6. 핵심 사용자 플로우(특히 ‘돈의 흐름’ 중심)

### 6.1 최초 1회: 기본 설정
1) 근무지 등록(이름/주소)
2) 급여 정보 등록: **기본급 1줄 + 단위 선택(시급/일급/월급)**
3) 송금 수신자 등록(허용 목록)
4) 홈에서 "내가 주로 무엇을 하려는지" 확인
   - 미리받기 / 월급 확인 / 가족 송금 / 보관

### 6.2 매일: 출근/퇴근(근무 증거 생성)
- 출근 탭 -> 기록 생성
- 퇴근 탭 -> 기록 확정
- (필요 시) 근무 시간 수정 -> **수정 사유 필수 + 추가 증거 선택 첨부**
- 앱은 기록 결과를 **"오늘 근무가 반영되면 어떤 금융 행동이 가능해지는지"**와 연결해 보여준다.

### 6.3 급여일 전: 미리받기
1) 반영된 근무일/시간 확인
2) 지금 가능한 금액 확인
3) 신청 금액 선택 -> 확인
4) 결과:
   - 가능: 신청 완료 + 상환 예정일 확인
   - 보완 필요: 미반영/확인 필요 기록 안내

### 6.4 매월: 급여 점검
1) 월간 요약 확인(근무일/근무시간/연장/야간)
2) 참고용 급여 추정 확인("참고/추정" 표기)
3) 급여일 이후 **실제 입금액 입력**
4) 결과:
   - 정상: "차이 없음(또는 허용 범위)"
   - 이상: "확인 필요한 차이" + "근거 제시"

### 6.5 차액 감지 후 액션(보호 기능)
차액 감지 화면에서 한 번에 제공:
1) **증빙 리포트 만들기**
2) **근거 자료 묶음 만들기**
3) **신고 준비(Instant Claim v0)**: 제출 경로 안내 + 자동 요약 문장 + 체크리스트 + 파일 공유 + 바로가기

> 핵심 UX 목표: 메인 가치가 금융이어도, 문제가 생기면 **근거 -> 문서 -> 다음 행동**이 같은 흐름 안에서 이어져야 한다.

### 6.6 송금(테스트넷)
1) 수신자 선택(허용 목록)
2) 금액 입력 -> 확인
3) 전송 후 상태 추적(SUBMITTED -> CONFIRMED/FAILED)
4) 영수증 PDF 생성 및 공유

### 6.7 보관/이자(데모 시뮬레이션)
1) 보관 가능한 잔액 확인
2) 예상 이자 확인
3) 필요 시 송금 또는 생활비로 전환

## 7. 기능 요구사항(상세)

## 7A. 금융 홈(Money Home)

### 핵심 원칙
- 첫 인상은 **핀테크**
- 홈은 "기록 화면"이 아니라 **"이번 달 내 돈 상태"를 보여주는 화면**
- 사용자가 지금 해야 할 행동을 **1개 CTA**로 강하게 제안

### H1. 메인 Hero 카드(P0)
- 제목은 **"이번 달 내 돈"** 기준으로 구성
- 필수 정보:
  - 예상 급여
  - 실제 입금 상태
  - 지금 미리받기 가능 금액
  - 지금 보낼 수 있는 금액
- 우측 또는 하단에 다음 행동 CTA 1개 노출

### H2. 상태 기반 추천 행동(P0)
- 급여 전: `미리받기 보기`
- 급여일 전후: `실제 입금액 입력`
- 차액 발생 시: `월급 확인하기`
- 급여 후: `송금하기`
- 여유 자금 존재 시: `보관하고 이자 보기`

### H3. 빠른 금융 액션(P0)
- 미리받기
- 가족 송금
- 보관/이자
- 월급 확인

### H4. 오늘 근무 카드의 역할 조정(P0)
- 홈에 남기되, 핵심 카드가 아니라 보조 카드로 둔다.
- 문구는 기록 그 자체보다 금융 연결을 강조한다.
  - 예: "오늘 근무가 반영되면 미리받기 한도가 늘 수 있어요"
  - 예: "퇴근 기록이 없으면 이번 달 반영이 지연될 수 있어요"

### H5. 상태 표현(P0)
- loading / empty / error / success 상태를 명시적으로 구분한다.
- 금액이 없는 경우에도 `왜 없는지`와 `다음 행동`을 함께 보여준다.

## 7B. 근무 증거(WorkProof)

### W1. 근무지 등록(P0)
- 이름, 주소, 지도 선택

### W2. 급여 정보 입력(P0, 단순 UI)
- 입력 UI는 **"기본급 1줄 + 단위 선택(원/시간, 원/일, 원/월)"**로 통일
- 단위별 조건 입력(1개만 추가 노출)
  - 일급: 하루 기준 근무시간(기본 8시간, 수정 가능)
  - 월급: 월 기준 근무시간(기본값 제공, 수정 가능)
- 내부 로직은 **시간당 기준으로 정규화(환산 시급)**하여 동일 엔진 재사용

### W3. 출퇴근 원탭 기록(P0)
- 출근/퇴근 시점에 위치 스냅샷 1회 저장(백그라운드 추적 없음)
- 기기 시간 + 서버 수신 시간 함께 기록(감사 로그 기반)

### W4. 근무 시간 수정(‘증빙 강화 UX’로 설계) (P0)
**목표:** "수정 가능 = 조작"이 아니라 **"수정 가능 = 설명 가능한 기록"**으로 신뢰 강화

#### W4-1 수정 사유 선택(필수)
- 사용자가 시간을 수정할 때 **수정 사유 선택이 필수**
- 예시(초기 제공):
  - 출근/퇴근 탭을 늦게 눌렀어요
  - 연장근무가 있었어요
  - 휴게시간이 달랐어요
  - 기록이 누락됐어요
  - 기타(직접 입력)

#### W4-2 추가 증거 첨부(선택)
- 사진(예: 작업지시서, 근무표, 현장 사진 등) / 메모 첨부 가능
- 첨부는 선택이며, "첨부 없음"도 명확히 기록

#### W4-3 수정 이력(감사 로그) 자동 저장
- 수정 시 아래 정보를 **자동으로 기록**
  - 누가: 사용자 ID(또는 계정)
  - 언제: 수정 시각(서버 시간)
  - 무엇을: 변경 전/후 시간(또는 변경된 필드)
  - 왜: 수정 사유(선택값 + 기타 메모)
  - 증거 유무: 첨부 개수/종류(사진/메모) 및 파일 참조
- 수정 이력은 **삭제 불가(감사 목적)**, 단 사용자는 "추가 메모"는 보완 가능(기록으로 남김)

#### W4-4 문서 자동 반영 명세
- 증빙 리포트/근거 자료 묶음에는 다음이 자동 포함되어야 함:
  - 수정된 기록 표시(배지)
  - 수정 이력 표(변경 전/후, 사유, 첨부 유무, 시각)
  - 첨부가 있는 경우: 첨부 목록(파일명/업로드 시각/해시 또는 식별값)

### W5. 월간 요약(P0)
- 월간 근무일, 총 근무시간, 연장/야간 시간 요약
- "수정된 기록 수" 요약 포함
- "반영 완료 / 확인 필요 / 미반영" 상태를 함께 표시해 금융 연결성을 높인다.

### W6. 금융 연결 상태(P0)
- 근무 화면은 단순 일지보다 **"금융 반영 현황"**으로 보이도록 구성
- 필수 표시:
  - 이번 달 반영 근무일
  - 확인 필요한 기록 수
  - 미리받기/월급 확인에 사용할 수 있는 기준일

## 7C. 미리받기(Advance, 데모 시뮬레이션)

### 핵심 원칙
- 미리받기는 P0에서 **전면 가치**로 보이되, 실제 상용 가불처럼 과장하지 않는다.
- 사용자는 **"왜 이 금액이 가능한지"**를 이해할 수 있어야 한다.

### A1. 가능 금액 계산(P0, 시뮬레이션)
- 반영된 근무분 × 환산시급 × 정책 비율로 계산
- 데모 기준 상한은 **최대 ₩50만**
- 근무 반영 상태에 따라 금액이 달라짐을 명시

### A2. 신청 UX(P0, 시뮬레이션)
- 추천 금액 칩 또는 직접 선택
- 예상 수령액 / 수수료 / 상환 예정일 표시
- 확인 후 신청

### A3. 근거 설명(P0)
- 아래 요소를 함께 표시:
  - 반영 근무일 수
  - 반영 근무시간
  - 확인 필요 기록 수
  - 다음 구간까지 얼마나 더 필요한지

### A4. 이력 관리(P0)
- 이번 달 신청 이력
- 누적 사용 금액
- 상환 예정일

### A5. 실제 상용화(P1)
- 실제 정책/파트너 연동
- 리스크 정책, 사용자 등급, 상환 정책 고도화

## 7D. 급여 점검(Wage Shield) — ‘정답 계산기’가 아니라 ‘이상 탐지’

### 핵심 원칙(문구/디스클레이머)
- 급여 계산 결과는 **"참고용 추정"**이며, 실제 지급/공제는 근로계약·법규·급여명세서에 따라 달라질 수 있다.
- 앱은 **이상 징후를 빠르게 찾고 근거를 보여주는 도구**이며, 법률 자문이 아니다.

### WS1. 참고용 급여 추정(P0)
- 입력된 급여 정보(시급/일급/월급 환산) + WorkProof 기반 시간 집계로 추정
- 중복 계산 방지(프리미엄은 추가분만)
  - 기본: 시급 × 총 근로시간 × 1.0
  - 연장 프리미엄: 시급 × 연장시간 × 0.5
  - 야간 프리미엄: 시급 × 야간시간 × 0.5
- 화면에는 항상 근거를 함께 표시:
  - 시간/구간/WorkProof ID
  - 수정 사유 유무(수정 배지)

### WS2. 실제 입금액 입력(P0)
- 사용자는 "통장 입금액(실수령)"을 입력하는 것을 기본으로 한다.
- 공제 항목을 입력하지 않은 경우:
  - **"공제 미반영" 배지** 표시
  - 알림 임계값을 완화(불필요 경보 최소화)

### WS3. 차액 감지(이상 징후 탐지) (P0)
- 임계값(기본값)
  - 차액 ≥ 30,000원 또는 예상액의 2%
  - 공제 미반영이면 50,000원 또는 3%로 완화
- 차액 화면에서 "가능한 원인"을 **근거와 함께** 제시
  - 예: 연장시간 합계, 야간시간 합계, 수정 기록 포함 여부

### WS4. 근거 중심 출력(P0)
- "총 차액"만 보여주지 않고, **항목별 근거**를 제공:
  - 연장/야간 합계 시간
  - 해당 WorkProof 목록(링크)
  - 수정 사유/첨부 유무 표시

### WS5. 문서 파싱 고도화(P1)
- 급여명세서/계약서 자동 추출(후속)
- MVP 단계에서는 수동 입력이 기본이며, 파싱은 "있으면 편한 기능"으로 제한

## 7E. 송금(Remittance, 테스트넷)

### R1. 수신자 허용 목록(P0)
- 별칭/관계/사진(선택) 포함
- 허용 목록 외 송금 불가
- "자주 보내는 사람" 표시 지원 권장

### R2. 송금 실행(P0)
- 테스트넷에서 스테이블코인(예: USDC 또는 모의 토큰) 전송
- 전송 전 반드시 사용자 확인(확인 화면)
- 송금 전 화면에는 아래를 명시:
  - 허용 목록 수신자
  - 데모 송금(실자금 이동 없음)
  - 영수증 저장 예정

### R3. 상태 추적(P0)
- 상태: SUBMITTED -> CONFIRMED / FAILED
- 실패 시 재시도 안내(정책 기반)
- 송금 상세 화면에서 상태와 전송 해시를 즉시 다시 열 수 있어야 함

### R4. 영수증(P0)
- 전송 해시(Tx Hash) 포함
- 영수증 PDF 자동 생성(문서 모듈 재사용)
- 문서 메인 목록보다 **송금 상세/내역에서 즉시 열 수 있는 경험**을 우선한다.

## 7F. 안전정책(SafePay)

- 허용 목록 기반 송금
- 수신자 변경 후 24시간 송금 제한(쿨다운)
- 고액 추가 확인(예: 평소의 2배 또는 특정 금액 이상)
- 중복 전송 방지(중복 방지 키)
- 차단 시에는 "왜 막혔는지 / 언제 다시 가능한지"를 사용자 언어로 설명해야 한다.

## 7G. 보관/이자(Vault, 데모 시뮬레이션)

**"남는 돈을 따로 보관하고, 필요하면 다시 송금할 수 있게"**

> 보관/이자는 DonDone의 메인 메시지를 해치지 않는 선에서,
> `남는 돈 보관 -> 예상 이자 확인 -> 송금 연결` 경험을 제공하는 데모 시뮬레이션 기능이다.

### 자금 순환 구조(데모 설명용)
```text
사용자 잔액
   ->
보관 금액 설정
   ->
예상 이자 계산(시뮬레이션)
   ->
필요 시 다시 송금/생활비로 전환
```

### V1. 보관 잔액 표시(P0, 시뮬레이션)
- 현재 보관 중인 금액
- 보관 가능 금액
- 송금 가능 전환 금액

### V2. 예상 이자 표시(P0, 시뮬레이션)
- 일별/월별 예상 이자
- 연 환산 기준값 표시
- 단, **수익 보장처럼 보이는 표현 금지**

### V3. 목표 저축(P1)
- 목표(학비/집 수리/귀국 자금) 설정 + 급여에서 자동 적립
- 진행률 바 + 예상 달성일
- 달성 시 "바로 송금" 옵션 연결

### V4. 실제 연동(P1)
- 메인넷 연동, 실 yield 파트너십 검토

---

## 7H. 문서(Documents): 증빙 리포트 / 근거 자료 묶음 / 신고 준비

### D1. 증빙 리포트(P0)
- 월 단위(또는 사건 단위)로 생성
- 필수 포함:
  - 월간 요약 + WorkProof 상세 표
  - 급여 추정 근거 + 차액 리포트
  - 수정 이력(감사 로그) 표
  - 첨부 목록(있으면)

### D2. 근거 자료 묶음(P0)
- 증빙 리포트 + 제출용 요약 + 체크리스트
- 출력 형태: PDF(기본) / ZIP(선택, 첨부가 많은 경우)

### D3. 신고 준비(Instant Claim v0) (P0)
- 자동 제출이 아니라 반자동 지원:
  - 접수 경로 안내(온라인/전화/방문)
  - 제출용 자동 요약 문장 생성(복사 버튼)
  - 근거 자료 묶음 공유(메일/메신저/저장)

## 8. 인공지능(코파일럿) 설계

### 8.1 목적: “자유대화”가 아니라 “현재 화면을 돕는 설명/문장 생성”
- 각 핵심 화면에 **"이 화면 설명해줘"** 버튼 제공
- 추천 질문 칩(예시)을 함께 제공해 사용자가 무엇을 물어볼지 고민하지 않게 한다.

### 8.2 코파일럿 UX(버튼/칩 예시)
#### (1) 금융 홈
- 버튼: `이 화면 설명해줘`
- 추천 질문 칩:
  - `지금 내가 할 수 있는 행동은 뭐야?`
  - `왜 미리받기 금액이 이렇게 나왔어?`
  - `보관과 송금 중 뭐가 맞아?`

#### (2) 차액 감지 화면
- 버튼: `이 화면 설명해줘`
- 추천 질문 칩:
  - `왜 차액이 생겼어?`
  - `어떤 근거로 그렇게 판단했어?`
  - `지금 내가 할 수 있는 다음 행동은 뭐야?`

#### (3) 송금 차단/보호 화면(SafePay)
- 버튼: `왜 송금이 막혔어?`
- 칩:
  - `24시간 쿨다운이 뭐야?`
  - `얼마부터 추가 확인이야?`
  - `다시 보내려면 어떻게 해?`

#### (4) 신고 준비 화면(Instant Claim)
- 버튼: `제출용 문장 만들어줘`
- 칩:
  - `짧게 요약해줘`
  - `정중한 문장으로 써줘`
  - `영어로 바꿔줘`

### 8.3 코파일럿이 제공하는 기능(권장 3종, P0)
> 코파일럿은 **설명/번역/문장 생성만** 수행한다.  
> 숫자 계산/판정은 서버 룰 기반 결과를 그대로 사용한다.

1) **돈 상태/차액/정책 차단 이유를 쉽게 설명**(근거 포함)
2) **제출/공유용 요약 문장 생성**
3) **화면 facts 기반 번역/쉬운 표현 변환**

### 8.4 가드레일(환각/오답 방지 정책) — PRD 고정
- 코파일럿은 **새로운 숫자를 계산하거나 추정하지 않는다.**
- 코파일럿은 서버가 제공한 **facts(사실 데이터)**만 사용해 문장을 만든다.
- facts 예시:
  - 차액 금액, 근무시간 합계, 연장/야간 시간, 해당 WorkProof ID 목록, 수정 사유 목록, SafePay 차단 코드, 허용 목록 상태, 미리받기 가능 금액 근거 등
- facts 밖의 내용을 말하려고 하면:
  - "확인할 수 없음/추정 불가"로 응답하고, 필요한 입력을 안내한다.
- 모든 코파일럿 응답에는 화면 하단에 공통 문구를 표시:
  - "이 안내는 화면에 표시된 사실을 바탕으로 작성된 참고 문장입니다."

### 8.5 RAG 정책: “필수 아님, 필요 시 RAG-lite”
- 대부분 질문은 화면 facts로 해결한다.
- 용어/기능 설명(예: 증빙 리포트 vs 근거 자료 묶음, 보관/이자 범위) 정도만 커버하기 위해:
  - 짧은 FAQ/용어집을 **키워드 검색 또는 소규모 임베딩 Top-k**로 제공(RAG-lite)
- 대규모 지식 기반 챗봇은 P1 이후로 미룬다.

## 9. 데모 전용 기능: 타임 트래블(Time Travel)

> **실사용 기능이 아니라 발표 데모를 위한 장치**이며, 릴리스 빌드에서는 비활성화한다.

### 9.1 목적
- 실제 한 달 사용 흐름(근무 -> 반영 -> 미리받기 -> 월간 요약 -> 급여 점검 -> 차액 감지 -> 송금/보관)을
  **15~30초로 압축**해서 보여주기

### 9.2 동작 방식(권장)
- 데모 시작 시 **시드(고정) 데이터**를 주입한다.
  - WorkProof 기록(일자별 출퇴근 + 일부 수정 이력 포함)
  - 미리받기 가능 금액 변화
  - 급여 이벤트(예상/실입금 입력)
  - 송금 이벤트(성공/실패 1건씩)
  - 보관/이자 시뮬레이션 이벤트
- 사용자는 화면 상단의 슬라이더/재생 버튼으로 `기준일(asOf: YYYY-MM-DD)`을 바꾼다.
- 앱은 기준일 이전 데이터만 반영해 다음 화면을 즉시 갱신한다.
  - 홈 상태, 월간 요약, 참고용 추정, 차액 감지, 추천 행동, 미리받기 가능 금액, 송금/보관 상태

### 9.3 구현 원칙(데모 품질을 위한 기준)
- **"시드 데이터는 고정"**이지만, **집계/계산/판정 로직은 실제 로직**을 사용한다.
- 기준일 변경은 "데모 모드 파라미터"로 서버에 전달한다.
  - 예: 요청 헤더 `X-Demo-AsOf: 2026-03-20`
- 결과적으로, 슬라이더만 움직여도 실제 사용자처럼 모든 화면이 자연스럽게 이어져야 한다.

### 9.4 데모 모드 정책
- 데모 전용 계정에서만 활성화
- 일반 계정/릴리스 빌드에서는 완전 비활성(설정 숨김)

#### (1) 블록체인 게이트웨이(remittance)

### 10.4 비동기 잡(jobs) 설계(송금/문서 생성 안정화)
- PDF 생성, 송금 전송/확정 확인은 "요청-응답"에서 분리한다.
- `jobs` 테이블로 상태/재시도를 관리한다.

#### 잡 상태
- QUEUED -> RUNNING -> DONE / FAILED

#### 잡 종류(예시)
- RENDER_PROOF_PACK
- RENDER_CLAIM_KIT
- SUBMIT_TRANSFER
- POLL_TRANSFER_RECEIPT
- RENDER_TRANSFER_RECEIPT

### 10.5 송금이 비동기 처리인 이유(요약)
- 전송은 빠를 수 있지만, **완료 확정은 전송 결과(영수증) 확인이 필요**
- 네트워크 재시도/중복 요청이 흔해 **중복 방지 키와 재시도 정책**이 필수
- nonce 충돌을 피하기 위해 **전송 잡은 직렬화(동시성=1)**가 MVP에 안전

### 10.6 데이터 모델(개요)
- users, workplaces, contracts
- workproof_records + workproof_modifications(수정 이력) + workproof_attachments
- advance_snapshots, advance_requests
- wage_estimates, wage_verifications
- recipients, transfers
- vault_deposits, vault_yield_logs
- documents(ProofPack/ClaimKit/Receipt) + document_links(관련 엔티티 연결)
- jobs
- events(감사 로그, append-only)

> 핵심: "수정 이력/첨부/근거"는 문서에 자동 포함되므로, DB에서도 분리 저장해 추적 가능해야 한다.

## 13. 3주 실행 계획(권장, Jira 기반)

### 13.0 인원 구성
- 풀스택: 강보승
- 풀스택: 차지훈
- 서버/인프라: 장현준
- 서버: 정관우
- 블록체인: 최성민
- 블록체인: 이건도

### 13.1 P0 포함 범위
- **WorkProof**: 출근/퇴근 기록, 근무 상세, 근무 수정(W4: 수정 사유 필수 + 선택 증거)
- **WageShield**: 월간 요약/예상(룰 고정), 입금 입력, 차액 감지(근거 포함)
- **Evidence**: Proof Pack/Claim Kit PDF 생성(수정 이력 포함)
- **Remit**: 테스트넷 송금, tx 상태 추적, 영수증 + SafePay 최소
- **Instant Claim v0**: 반자동 제출 준비(체크리스트/문장/경로 안내)
- **Demo Time Travel**: seed + asOf로 상태 렌더(데모용)
- **Chat Copilot(P0)**: "이 화면 설명해줘" + 추천 질문 칩(화면 facts 기반)

> 주의: 이전 안에 있던 `사업주 메시지`는 현재 제품/UX 기준에서는 **핵심 데모 동선보다 우선순위가 낮다.**
> Jira에는 `stretch` 또는 `optional` 라벨로 별도 관리할 수 있다.

### 13.2 Jira 운영 원칙

#### 이슈 계층(권장)
- **Epic**: 기능 축 단위
- **Story**: 사용자 흐름 또는 기능 단위
- **Sub-task**: 사람별 세부 작업

> 권장 원칙: **Epic은 기능 기준**, **Story는 사용자 흐름/기능 기준**으로 관리한다.  
> 즉, Epic에 `[ANDROID]`, `[BACKEND]`, `[BLC]`를 직접 붙이기보다  
> Epic은 `[WRK]`, `[WAGE]`, `[DOC]`, `[BLC]`, `[INFRA]`처럼 **기능/영역 기준**으로 만들고,  
> Story는 prefix 없이 **무엇을 할 수 있는지 / 무엇을 만든는지**가 바로 보이게 작성하는 편이 추적에 유리하다.

#### Prefix 규칙(권장)
- **Epic prefix**: 기능/도메인 기준, 대문자 고정
  - 예: `[AUTH]`, `[WRK]`, `[ADV]`, `[WAGE]`, `[DOC]`, `[BLC]`, `[INFRA]`, `[COP]`
- **Story 제목**: prefix 없이 사용자 관점 또는 기능 관점으로 작성
  - 예: `Jenkins 빌드 파이프라인을 구성한다`
  - 예: `사용자는 급여 차이를 확인할 수 있다`
  - 예: `근거 자료 묶음을 생성하고 공유할 수 있다`
- **Sub-task 제목**: 기술 담당 기준 prefix 사용 가능
  - 예: `[ANDROID] 금융 홈 Hero 카드 UI 구현`
  - 예: `[BACKEND] 급여 차이 계산 API 구현`
  - 예: `[INFRA] Jenkinsfile 작성`

#### 워크플로우(권장)
- `Backlog -> Selected for Dev -> In Progress -> In Review -> Done`
- 장애/대기 상황은 `Blocked` 라벨로 관리

#### 라벨(권장)
- `p0`, `p1`, `frontend`, `backend`, `infra`, `blockchain`, `demo`, `copilot`, `pdf`, `time-travel`, `safepay`, `stretch`

#### Definition of Done(권장)
- 프론트: loading / empty / error / success 상태 중 필요한 상태가 보인다.
- 서버: API DTO/validation/logging이 정리되어 있다.
- 블록체인: tx hash / 상태 / 실패 사유가 재현 가능하다.
- 데모: seed/reset/asOf로 재현 가능하다.
- 문서: 생성/다운로드/링크 확인이 가능하다.

### 13.3 Jira Epic 구조(권장)

| Epic | 범위 | 주 담당 |
|---|---|---|
| `[WRK] 근무 기록` | 출근/퇴근, 상세, 수정(W4), 근무 집계 | 풀스택 + 서버 |
| `[WAGE] 급여 점검` | 월간 요약, 예상, 입금 입력, 차액 감지 | 풀스택 + 서버 |

  - WorkProof API/DB(근무 생성/조회)
  - 월간 집계/계산 엔진 스켈레톤(총시간/야간/연장 구분 정도)
- **최성민(블록체인)**
  - 지갑 연결 + 테스트넷 전송 PoC(tx 생성/서명/전송)
  - tx 상태 확인 방식 결정(우선 폴링)
- **이건도(블록체인)**
  - 무결성 해시 규격 초안(해시 대상 필드/메타 구조)
  - 데모 시드 데이터 포맷(JSON) 초안(정상 케이스 1)

### 13.5 2주차: 차액 감지 -> Proof Pack -> 송금/영수증 연결 + 근무 수정(W4)

#### 주말까지 완료 기준
- 타임트래블로 급여일까지 이동 -> 입금 입력 -> 차액 감지 -> Proof Pack 생성 -> 송금 -> 영수증까지 한 번에 된다.

#### 사람별 작업
- **강보승(풀스택)**
  - 입금 입력 UX + 차액 결과 화면(원인 카드/근거 링크)
  - 차액 이후 액션 UI(Proof Pack / Claim 준비 / 신고 준비)
  - SafePay 관련 UI(차단/추가 확인/안내 화면)
- **차지훈(풀스택)**
  - Proof Pack 생성 UX(진행/완료, "빠르게 생성" 연출)
  - 영수증 화면(Explorer 링크/tx hash/상태) + 공유 버튼
- **장현준(서버/인프라)**
  - Proof Pack PDF 생성 MVP(근무 요약 + 생성 메타)
  - 파일 저장/다운로드 경로 구성(간단 스토리지 또는 URL)
- **정관우(서버)**
  - 근무 수정(W4): 수정 사유 필수 + 선택 증거 메타 저장
  - 차액 감지 로직 + 사유(룰 기반) + 근거 DTO(WorkProof 리스트/표)
  - `/demo/seed`, `/demo/state?asOf=` 구현(시드 주입 + 날짜 기준 집계)
- **최성민(블록체인)**
  - 송금 E2E(전송 -> 상태 -> 영수증 데이터)
  - SafePay 최소 적용(allowlist/쿨다운/고액확인/idempotency)
- **이건도(블록체인)**
  - Proof Pack 해시 생성(오프체인) + QR payload 확정
  - 데모 시드 "차액 케이스" 1개 추가(의도된 누락/차액)

### 13.6 3주차: Instant Claim v0 + 챗봇(P0) + QR 검증 + 안정화

#### 최종 완료 기준
- PRD 6-step 데모 완주
- QR 검증 화면에서 `OK`
- 코파일럿이 "왜 차액?"를 facts-only로 설명한다.
- Instant Claim은 반자동(사용자 제출) 문구가 고정되어 있다.

#### 사람별 작업
- **강보승(풀스택)**
  - Instant Claim v0 화면 플로우(유형 선택 -> 키트 생성 -> 경로 안내)
  - 코파일럿 UI("이 화면 설명해줘" + 추천 칩 5개)
  - 데모 동선(타임트래블 -> 차액 -> 팩 -> 송금 -> 클레임 -> QR/챗봇 하이라이트) 고정
- **차지훈(풀스택)**
  - 제출 체크리스트/안내/문구 폴리싱(반자동 강조)
  - 영수증/문서/검증 화면 폴리싱(시각 임팩트)
- **장현준(서버/인프라)**
  - E2E 테스트 2개(정상/차액) + 버그 배시 운영
  - 데모 모드 재현 스크립트(seed/reset 한 번에)
- **정관우(서버)**
  - Claim Kit 생성(템플릿 v1: 일자별/수정사유 포함 + 무결성 섹션 포함)
  - 코파일럿 API 2개 핵심:
    - `/ai/explain` (차액/정책 사유 설명)
    - `/ai/claim-summary` (제출용 요약 문장)
  - facts-only 가드레일: "숫자 새로 계산 금지 / 서버 facts만 사용"
  - `employer-message`는 필요 시 `stretch`로 별도 관리
- **최성민(블록체인)**
  - 실패 케이스 2개 고정(거절/네트워크 등) + 상태 표준화
  - SafePay 차단 사유 코드/로그 정리(프론트 표시와 연결)
- **이건도(블록체인)**
  - QR Verifier 웹(검증 OK/FAIL, 생성일/포함 건수/해시 표시)
  - 발표용 검증 성공 장면 스크립트/데이터 고정

### 13.8 Jira 운영 팁
- Epic은 기능 축으로 끊고, Story는 prefix 없이 사용자 흐름이나 기능 단위로 쓴다.
- `[ANDROID]`, `[BACKEND]`, `[BLC]`, `[INFRA]` prefix는 Sub-task에서만 쓰는 것을 권장한다.
- Story 하나는 가능하면 **기획적으로 닫히는 단위**로 두고, 구현 분리는 Sub-task로 내린다.
- `Blocked` 사유는 반드시 외부 의존성(API, RPC, PDF, env) 중 무엇인지 명시한다.
- 매주 말에는 `데모 완주 기준`으로 Done 여부를 판단한다.
