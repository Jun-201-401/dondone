# App/Web Shared Policy

## 목적
- worker 앱과 employer 웹이 같은 용어와 같은 정책을 보도록 고정한다.
- Slice 6에서는 현재 구현된 범위를 기준으로 정책을 정리하고, 미구현 lifecycle은 후속으로 남긴다.

## 현재 고정 용어
- `정정 요청`
  - 출퇴근 시간 변경이 필요한 경우 근로자가 보내는 요청이다.
  - employer 웹에서는 승인/반려 대상이다.
- `검토 필요 기록`
  - 시스템이 `WorkProof`를 `NEEDS_REVIEW`로 분류한 기록이다.
  - 반경 밖 퇴근, review flag 같은 예외 기록이 여기에 해당한다.
- `기록 검토`
  - `검토 필요 기록`을 사람이 확인하는 정책 용어다.
  - 현재는 조회 전용이며, `검토 확인/검토 반려` command는 아직 없다.
- `첨부 파일`
  - 기본 출퇴근 입력값이 아니라 정정 요청 또는 기록 검토 설명을 보강하는 증빙이다.

## 현재 구현 상태

### Worker 앱
- checkout 응답에서 `reflectionStatus`, risk flag를 통해 검토 필요 여부를 알 수 있다.
- 시간 정정 요청은 별도 correction flow로 이어질 수 있다.
- 반경 밖 퇴근 등 `검토 필요 기록`에 대해 별도 `기록 검토 요청` command는 아직 없다.

### Employer 웹
- `정정 요청`은 목록/상세/승인/반려가 가능하다.
- `검토 필요 기록`은 이슈 큐와 상세 조회가 가능하다.
- `검토 필요 기록`은 아직 read-only이며 `검토 확인/검토 반려` command는 없다.

## 화면 표현 원칙
- `정정 요청`과 `검토 필요 기록`은 같은 이슈 화면에서 보더라도 성격이 다르다는 점을 분리해서 표시한다.
- `정정 요청`에는 원본 시간, 요청 시간, 요청 사유, 승인/반려 결과를 보여준다.
- `검토 필요 기록`에는 검토 사유, 기록 상태, 반영 상태, 위치/사업장 정보 같은 review 문맥을 보여준다.
- 첨부 파일은 우선 `증빙 메타데이터`로 취급한다.

## 이번 Slice 6에서 고정하지 않는 항목
- worker 앱의 `기록 검토 요청` 제출 command
- employer 웹의 `검토 확인 / 검토 반려` command
- 첨부 파일 인앱 미리보기/다운로드
- `late / leave / absent`
- invitation token을 대체하는 가입 코드 운영 flow

## 후속 방향
- worker 앱이 `검토 필요 기록`에 설명/첨부를 추가할 수 있는지 별도 정책 확정이 필요하다.
- employer 웹은 이후 `검토 확인 / 검토 반려`와 audit 필드를 도입할 수 있다.
- 가입 onboarding은 현재 invitation token 기준이며, 이후 회사 가입 코드 기반으로 단순화할 수 있다.
