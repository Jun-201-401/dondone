# Problem And Goal

## 문제
현재 사용자는 송금이 느리다고 느끼고 있다. 체감상 약 7초 정도 걸린다고 말하고 있지만, 아직 이 숫자가 항상 재현되는지, 그리고 정확히 어디서 시간이 쓰이는지는 확인되지 않았다.

송금 지연은 한 가지 원인으로 생기지 않을 수 있다.

- 서버가 `POST /api/remittance/transfers` 를 처리하는 시간
- 블록체인 RPC 응답 시간
- 비동기 worker 대기 시간
- receipt 확인 시간
- 모바일의 reload / polling 대기 시간

그래서 이번 작업은 "느리다"를 바로 고치는 대신, 먼저 "정말 느린가"와 "어디가 느린가"를 분리해서 확인하는 방식으로 진행한다.

## 목표
이번 작업의 목표는 아래 네 가지다.

1. 실제 지연을 숫자로 확인한다.
2. 총 시간을 여러 구간으로 나눠 가장 큰 병목을 찾는다.
3. 가장 큰 병목부터 작은 리스크로 줄인다.
4. before/after 결과를 포트폴리오에 그대로 쓸 수 있게 남긴다.

## 성공 기준
- 최소 한 개 핵심 지표에서 명확한 before/after 개선을 만든다.
- 개선 과정이 처음 보는 사람에게도 설명 가능해야 한다.
- 같은 사용자 동시 송금 안전성, 정책 정확성, 비동기 상태 추적 안정성을 깨지 않는다.

## 지금 당장 답해야 하는 질문
- 사용자가 말한 7초는 실제로 재현되는가
- 그 시간은 request path, worker path, mobile path 중 어디에 가장 많이 쓰이는가
- `demo` 와 `sepolia` 에서 병목 구조가 같은가 다른가
