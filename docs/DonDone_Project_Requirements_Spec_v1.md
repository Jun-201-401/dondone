| 항목 | 본 문서 기준 |

## 1. 목적
- PRD v1.5의 P0 요구를 구현 가능한 수준으로 압축 정리한다.
- 팀 내 역할(기획/개발/테스트/리뷰)이 공통으로 참조할 기준 문서를 제공한다.
- 데모 시연 기준과 제품 제약(testnet/demo only, evidence-first)을 명확히 고정한다.

## 2. 제품 범위

### 2.1 포함 범위 (P0)
- 근무 기록 기반 금융 홈(Money Home)
- 인증/권한(Auth)
- 근무 증거 생성/관리(WorkProof)
- 미리받기(Advance, 데모 시뮬레이션)
- 급여 점검(Wage Shield, 이상 탐지+근거 제시)
- 문서 생성/관리(Documents)
- 정산 문제 해결 보조(Instant Claim v0)
- 테스트넷 송금(Remittance)과 사전 정책(SafePay)
- 보관/이자(Vault, 데모 시뮬레이션)
- facts 기반 코파일럿(Copilot)
- 데모 타임트래블(Demo Time Travel)
- 고용주 지원축(회사 확인 코드/운영 콘솔/요약 카드)

### 2.2 제외 범위
- 실거래/실정산/실제 수익 보장
- 메인넷 전환 및 실금융 파트너 심사 정책
- Copilot 자유 대화형 고도화(RAG 확장)
- P1 기능(정교한 자동화, 고도화 추천/알림)

## 3. 사용자 및 이해관계자
- 메인 사용자: 외국인 근로자(근무 기록-금융 행동 연결 필요)
- 보조 사용자: 고용주/운영 담당자(운영 리스크 가시화 필요)
- 내부 이해관계자: 모바일/백엔드 개발, QA, 데모 발표 담당

## 4. 기능 요구사항

### 4.1 공통 요구사항
- `FR-COMMON-001` 제품명은 `DonDone`으로 표기해야 한다.
- `FR-COMMON-002` 모든 P0 기능은 `demo/testnet only` 제약을 명시해야 한다.
- `FR-COMMON-003` 보호 API는 인증 사용자 소유 데이터만 접근 가능해야 한다.
- `FR-COMMON-004` 화면 상태는 `loading/empty/error/success`를 명시적으로 구분해야 한다.
- `FR-COMMON-005` 급여 결과는 최종 법률/재무 판단이 아니라 참고용 이상 탐지로 표시해야 한다.

### 4.2 Auth
- `FR-AUTH-001` 사용자는 회원가입/로그인으로 액세스 토큰을 발급받아야 한다.
- `FR-AUTH-002` 보호 기능 진입 전 `me` 조회로 사용자 컨텍스트를 복구할 수 있어야 한다.
- `FR-AUTH-003` 공개 경로는 최소 범위로 제한해야 한다.

수용 기준:
- 비인증 상태에서 보호 API 접근 시 인증 오류를 반환한다.
- 인증된 사용자는 본인 컨텍스트(`id`, 역할, 기본 프로필)를 조회할 수 있다.

### 4.3 Money Home
- `FR-HOME-001` 홈은 "지금 쓸 수 있는 돈" 중심 요약을 제공해야 한다.
- `FR-HOME-002` 홈은 다음 행동 CTA 1개를 우선 노출해야 한다.
- `FR-HOME-003` 빠른 액션으로 Advance/Wage/Remittance/Vault 진입을 제공해야 한다.
- `FR-HOME-004` 값이 비어도 원인과 다음 행동을 함께 안내해야 한다.

수용 기준:
- 홈 진입 시 금융 중심 히어로 카드가 최상단에 노출된다.
- 데이터 부재 시 단순 `0` 표시가 아니라 설명 문구+행동 유도가 표시된다.

### 4.4 WorkProof
- `FR-WP-001` 사용자는 근무지/급여 계약을 등록할 수 있어야 한다.
- `FR-WP-002` 출근/퇴근 기록을 생성/확정할 수 있어야 한다.
- `FR-WP-003` 누락/오류 기록에 대해 provisional 생성 또는 수정 요청이 가능해야 한다.
- `FR-WP-004` 월간 목록/상세/요약을 조회할 수 있어야 한다.
- `FR-WP-005` WorkProof integrity 근거(시간/위치/수정 이력)를 보존해야 한다.

수용 기준:
- 출근/퇴근은 필수 입력 검증을 통과한 경우에만 반영된다.
- 월간 요약은 반영 상태 구분(기록/반영/검증/선지급 가능)을 제공한다.

### 4.5 Advance (데모 시뮬레이션)
- `FR-ADV-001` 반영 근무 기반 가능 금액을 조회할 수 있어야 한다.
- `FR-ADV-002` 사용자는 가능 금액 범위 내에서 신청할 수 있어야 한다.
- `FR-ADV-003` 신청 결과와 상환 예정 정보를 조회할 수 있어야 한다.
- `FR-ADV-004` 보완 필요 시 사유(미반영/검증 필요)를 설명해야 한다.

수용 기준:
- 가능/불가 판단 사유가 사용자에게 설명 가능한 형태로 노출된다.
- 문구상 실대출/실정산으로 오해되는 표현을 사용하지 않는다.

### 4.6 Wage Shield (보호 흐름)
- `FR-WAGE-001` 이번 달 반영 근무 기준 예상 금액을 제시해야 한다.
- `FR-WAGE-002` 실제 수령 금액 입력/확인을 받아 차이를 계산해야 한다.
- `FR-WAGE-003` 차이가 발생하면 "확인 필요 상태"와 근거를 제공해야 한다.
- `FR-WAGE-004` 결과는 참고용 추정임을 명시해야 한다.

수용 기준:
- 정상/확인필요 상태가 구분되어 표시된다.
- 차이 설명은 근무/공제/입금 정보 근거와 함께 제시된다.

### 4.7 Documents / Instant Claim
- `FR-DOC-001` 증빙 리포트, 근거 자료 묶음, 송금 영수증 문서 생성을 지원해야 한다.
- `FR-DOC-002` 문서 생성은 비동기 처리로 진행되어야 한다.
- `FR-DOC-003` Instant Claim v0는 자동 제출이 아니라 반자동 준비 흐름이어야 한다.
- `FR-DOC-004` 문서는 다운로드/공유 가능한 상태를 제공해야 한다.

수용 기준:
- 문서 요청 후 처리 상태(대기/완료/실패)를 조회할 수 있다.
- 실패 시 재시도 또는 안내 메시지를 제공한다.

### 4.8 Remittance / SafePay
- `FR-REM-001` 송금은 등록된 수신자 허용 목록 기반으로만 가능해야 한다.
- `FR-REM-002` 전송 상태를 추적할 수 있어야 한다.
- `FR-REM-003` 송금 요청은 중복 방지 키를 지원해야 한다.
- `FR-REM-004` SafePay 정책(한도/지연/차단)을 사전에 평가해야 한다.
- `FR-REM-005` 테스트넷 데모임을 명확히 고지해야 한다.

수용 기준:
- 동일 idempotency 키 재요청 시 중복 송금이 발생하지 않는다.
- 상태 화면에서 최소 `SUBMITTED/CONFIRMED/FAILED`를 구분한다.

### 4.9 Vault (데모 시뮬레이션)
- `FR-VAULT-001` 보관 잔액과 예상 이자를 조회할 수 있어야 한다.
- `FR-VAULT-002` Vault와 송금 가능 금액의 관계를 설명 가능해야 한다.
- `FR-VAULT-003` 수익 보장처럼 해석될 수 있는 표현을 금지해야 한다.

수용 기준:
- 예상 이자 값은 데모 시뮬레이션 표기로 노출된다.
- 실투자/실수익 확정 문구가 노출되지 않는다.

### 4.10 Copilot (facts-only)
- `FR-CO-001` 화면 facts 기반 설명/요약/번역을 제공해야 한다.
- `FR-CO-002` 숫자 재계산/추정 답변을 기본 금지해야 한다.
- `FR-CO-003` 정책 위반 요청 시 안전 응답으로 전환해야 한다.

수용 기준:
- Copilot 응답은 화면 데이터와 추적 가능한 근거를 벗어나지 않는다.
- 불확실한 내용은 단정 대신 확인 필요 안내를 제공한다.

### 4.11 Demo Time Travel
- `FR-DEMO-001` 데모 계정에서 `asOf` 기준 조회를 지원해야 한다.
- `FR-DEMO-002` 홈/근무/미리받기/급여/송금/보관 읽기 흐름에 재현 상태를 반영해야 한다.
- `FR-DEMO-003` 일반 릴리스 사용자 흐름에서는 비활성화되어야 한다.

수용 기준:
- 데모 타임트래블 활성 시 동일 계정에서 일관된 시점 재현이 가능하다.
- 운영/실사용 계정에는 노출되지 않는다.

### 4.12 Employer Support (P0 지원축)
- `FR-EMP-001` 회사 확인 코드 기반 연결을 지원해야 한다.
- `FR-EMP-002` 운영 콘솔에서 누락/수정/확인 필요 상태를 볼 수 있어야 한다.
- `FR-EMP-003` Verified Worker Summary를 통해 운영 리스크 신호를 확인할 수 있어야 한다.
- `FR-EMP-004` 근로자 보호 흐름을 침해하지 않는 읽기/확인 중심 접근이어야 한다.

수용 기준:
- 고용주 기능이 근로자 중심 흐름을 대체하지 않고 지원축으로 동작한다.
- 과도한 감시/판정 시스템처럼 보이는 UX 문구를 피한다.

## 5. 데이터/계약 요구사항
- `CR-001` 요청 DTO는 Bean Validation 수준의 필수/형식 검증이 가능해야 한다.
- `CR-002` 응답은 공통 envelope 규약(`data` 중심)을 유지해야 한다.
- `CR-003` 문서 생성/송금 계열은 비동기 상태 조회 계약을 제공해야 한다.
- `CR-004` 리소스 소유권 위반 시 상세 노출 없이 은닉 전략을 유지해야 한다.
- `CR-005` 본 문서는 상위 요구사항 기준이며 endpoint/필드 상세는 API 계약 문서를 우선한다.

## 6. 보안 요구사항
- `SEC-001` 보호 API는 JWT 인증을 필수로 한다.
- `SEC-002` 공개 엔드포인트와 JWT 제외 경로는 명시적으로 관리되어야 한다.
- `SEC-003` 사용자 간 데이터 격리를 보장해야 한다.
- `SEC-004` 민감정보(토큰/개인키/비밀값)는 저장/로그/응답 노출을 금지해야 한다.
- `SEC-005` 외부 연동(체인/PDF/스토리지/AI)은 어댑터 인터페이스 뒤에서 통제되어야 한다.

## 7. 비기능 요구사항
- `NFR-001` 홈 주요 요약은 사용자 체감 기준으로 지연 없이 조회 가능해야 한다.
- `NFR-002` 송금/문서 생성 비동기 흐름은 상태 추적 가능해야 한다.
- `NFR-003` 오류 응답은 사용자 행동 가능한 메시지를 포함해야 한다.
- `NFR-004` 감사/추적을 위해 핵심 이벤트(신청/생성/전송/상태변경)를 로그화해야 한다.
- `NFR-005` 모바일 UI는 핵심 화면에서 상태 분기(loading/empty/error/success)를 제공해야 한다.
- `NFR-006` 문구/디스클레이머는 PRD 원칙(evidence-first, demo-only)을 유지해야 한다.

## 8. 제약 및 정책
- 본 제품 P0는 테스트넷/데모 시연 범위만 다룬다.
- 실제 금융 판단/대출 심사/법률 결론을 자동화하지 않는다.
- 급여 결과는 최종 판정이 아닌 확인 보조 흐름으로 제공한다.
- 고용주 기능은 근로자 보호 흐름 보조를 목적으로 한다.

## 9. 수용 기준 (데모 레벨)
- `AC-001` 3분 데모 내 핵심 플로우 완주:
  - 근무 기록 -> 미리받기/송금 -> 급여 점검 -> 문제 시 문서/클레임 준비
- `AC-002` 홈 첫 인상은 근무 기록 앱이 아니라 금융 홈으로 인지되어야 한다.
- `AC-003` 급여 이상 케이스에서 근거 기반 다음 행동이 끊기지 않아야 한다.
- `AC-004` 송금/문서 생성은 상태 추적과 재시도 안내를 제공해야 한다.
- `AC-005` 모든 시연 흐름에서 demo/testnet 디스클레이머가 유지되어야 한다.

## 11. 열린 이슈
- P0 요구사항 문서를 API 계약 수준(필드 단위)까지 확장할지 여부
- P1 요구를 본 문서의 후속 버전(v2)로 통합할지 여부
- 다국어 문구 기준(한국어/영어/사용자 언어) 우선순위 확정 필요

## 12. 가정
- 사용자 요청의 "요구사항 명세서"는 P0 기준의 실무용 통합 문서 초안을 의미한다.
- 구현 세부 계약은 기존 API 문서가 단일 진실원(Source of Truth)으로 유지된다.
- 본 문서는 합의/리뷰/테스트 기준 문서이며 코드 변경을 포함하지 않는다.
